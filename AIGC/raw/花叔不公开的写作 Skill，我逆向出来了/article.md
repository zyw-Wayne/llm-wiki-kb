# 花叔不公开的写作 Skill，我逆向出来了
从花叔的四本橙皮书和开源仓库出发，逆向拆解他的写作框架。方法、发现、架构、案例，全部公开。

作者 | Zerox在探索编辑 | Zerox在探索

花叔出了三本技术橙皮书。Claude Code、OpenClaw、Harness Engineering、Hermes Agent。都是那种你打开就能用的类型。速度很快，结构很稳，风格高度一致。

但你知道他最厉害的不是写了什么书，是他写书的方法没开源。

我记得他自己也在公众号评论区说过，这个技能短期内不打算开源。

出于好奇和学习的态度，所以我就打算自己干了。从他已经开源的 skills 和这四本书本身的结构、风格出发，一点一点拆。看他的写作框架到底长什么样。拆完之后我做了一个事：

把这个逆向出来的创作 Skill 开源了，还用它写了一本完整的 DeerFlow 橙皮书。

今天这篇文章，把整个逆向过程、发现、架构和用法都讲清楚。文末有 GitHub 链接，你可以直接拿去用。

## 花叔的橙皮书好在哪
我第一次读花叔的书，最大的感受是：这不是那种你看了开头就知道结尾的技术文档。

他的书有几个很明显的特征。

结构非常清晰。Part 1 起步，Part 2 核心能力，Part 3 进阶实战。读者读完一章能做一件事，不是看完了还在原地。

风格高度统一。几本书读下来，你会感觉是同一个人写的。短句。第一人称。先结论后展开。每章开头有具体时间线。不是"在当今这个 AI 时代"那种水开头。

写作的规则很明确。单句不超过 25 字。有具体数字。有禁用词表。写完了还要过 12 项章节 QC 和 10 项全书 QC。

> 这种一致性绝对不是靠灵感维持的，它背后一定靠的是一套系统。
而这套系统，花叔还没有开源。

## 我为什么要逆向
有几个原因。

一个是我真的想学。花叔这套方法论很强。从他的书里能看出他对技术写作的理解比我深很多。作为一个在 AI 领域折腾的人，我觉得搞清楚他的方法，比单纯读他的一本书有价值得多。

另一个是好奇。他的 skill 开源了很多个，最近的一次是我非常喜欢的nuwa-skill，女娲技能，让你体验造人的快乐（bushi。但橙皮书写作的 skill 没放出来，而且这也是花生老师的短期的内容输出护城河，不开源是很有必要的。

但我也想用怎么办，所以我就在想，能不能从他开源的仓库和橙皮书中挖一挖，反推出那套方法论。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/8R1W0bhIoib4LTwpDIJnMSsF8q3KSzlnovVzYNYzQJZ5Yia1NEkA4nv1ZjnNPgezZ5l1hWic7pDRhicPmP9Gw33qSnw4w717RYeVVzzfmibd0jF8/640?wx_fmt=png&from=appmsg&watermark=1)

思路其实不复杂。已开源的 skills 是工具，已出版的书是产物。拿着产物倒推工具，再跟已有的工具交叉验证，应该能还原出大概。

## 我发现了什么
这个过程比我想的有意思。拆开之后我发现了五层东西。

### 第一层：渐进式信任建立
花叔每本书、每章的开头都有时间线锚点。不是泛泛而谈的"我用了很久"，是具体的"用了 3 个月 Cursor 之后切换到 Claude Code"。

这个设计的核心是建立可信度。读者看到开头就知道，这个人真的用过这个东西，他不是拿文档在编。

逆向出来的 Skill 里保留了这个规则：每章前 2-3 段必须有具体时间线锚点或场景还原。没体验就诚实标注，不编造。

### 第二层：结构化的知识传递路径
几本书都遵循同一个递进逻辑：Part 1 从零到第一次跑通，Part 2 深入核心能力，Part 3 多场景实战。

这不是随便分的。每章解决一个具体问题，读者读完一章能做一件事。章节之间有向前桥接，不让读者停在某处不知道接下来该看什么。

### 第三层：风格 DNA
这是我拆解过程中最有收获的部分。

花叔的写作不是"写得好看"，是写得精确。有几条规则我直接抄过来：

- 单句不超过 25 字，用句号断句
- 第一人称高频出现，每千字至少 3 次
- 具体数字，不用模糊量词
- 禁用词表：不用"综上所述"、"值得注意的是"、"接下来我们将"
- 有体验就断言，没体验就诚实标注
最让我惊讶的是禁用词表。是明确的禁用词和替代词的对照表。比如"进行操作"换成"点击"或"输入"，"实现功能"换成"做到"或"搞定"。

> 风格一致性不靠天赋，是靠规则。

### 第四层：TDD 式质量铁律
这个最狠。

每章写完后要过 12 项 QC 检查：结构、风格、内容三块。全书写完再过 10 项全局 QC。不通过就重写。不妥协。

更绝的是有个叫 Rationalization Table 的东西——专门对付 Agent 找借口跳过规则的。比如 Agent 会说"开头不重要，内容才是关键"，表里直接回："花叔风格的核心就是开头。没有锚点的章节不可信。"

这个设计太聪明了。不让AI随意发挥，那就要给 AI 画线。

### 第五层：Agent Protocol
写作前必须先研究确认事实。版本信息、API 是否存在、社区反馈。不凭训练数据编造。

这一步保证了内容不是训练数据的幻觉，是真正经过验证的事实。

## 架构设计
我把逆向出来的系统拆成了2个子技能。

### 为什么要拆成两个 Skill
原始的花叔方法论里，内容创作和格式转换是混在一起的。但我做的时候发现一个问题——创作 Skill 本身已经很复杂了，8 个参考文件、3 个脚本、2 个模板、2 套测试。如果再把 PDF/HTML 导出的逻辑塞进去，这个 Skill 就太重了。

所以我做了一个架构决策：创作 Skill 只管内容生成，格式转换单独做成一个叫**md2book**的 Skill。

创作 Skill 把书的内容变成一个 Markdown 文件。md2book 负责把这个 Markdown 转成 PDF 或 HTML。两个 Skill 各司其职，用的人也清楚自己该装哪个。

### 项目结构

- SKILL.md— 主入口：心智模型、工作流、表达 DNA
- references/— 8 个参考文件：书籍蓝图、章节模板、风格 DNA、开头技巧、内容块模式、QC 清单、Agent 协作流程、调研来源
- assets/— 封面模板、目录模板、大纲 Schema
- scripts/— PDF 导出、结构验证、Agent 调度
- tests/— 基线检查 + 压力测试

### 质量检查机制
这个部分我单独拎出来说，因为它是整个系统最不像 AI 工具的部分。

一般的 AI 工具生成完就结束了。但这个 Skill 要求生成后必须过 QC。12 项章节检查加 10 项全书检查，每项都是 checkbox，必须逐项过。

还有前面提到的 Rationalization Table，不只是检查内容质量，还是一个严格检查执行纪律的规范。Agent 想偷懒？不行。表里写好了每个借口的反驳。

> 输出不能"差不多就行"，必须"通过了才放行"。

## 实践验证：DeerFlow 橙皮书
光逆向没用。得验证这套方法能不能真正产出一本书。

所以我选了字节刚开源的 DeerFlow 2.0 做实验。

DeerFlow 是字节的 Super Agent Harness，2026 年 2 月发布，当天就上了 GitHub Trending 第一名。技术含量够，社区讨论够，用来测试逆向 Skill 的实战能力刚好。

用这个逆向 Skill 写出来的 DeerFlow 橙皮书，结构是这样的：5 个 Part，10+ 节，从"DeerFlow 是什么"一路讲到架构、部署、核心功能、进阶扩展。

> DeerFlow 的定位就是这样：它是一个套在 Agent 身上的装备，让 Agent 能干活、能完成任务的完整系统。
短句。第一人称。类比解释。时间线锚点。禁用词零出现。

还原度我自己评估在 80% 以上。结构完全还原，风格细节有几处还差一点，毕竟我没见过花叔的原始 Skill，全靠逆向。但作为一个能跑通的系统，我觉得及格了。

DeerFlow 橙皮书的完整 Markdown 和 PDF 版本，我也把它们都跟着一起公开。有兴趣的大家品味一下。

## 怎么用
两个仓库，都很简单。

### 橙皮书创作 Skill
git clone https://github.com/ZeroxZhang/huashu-bookwriter
cp -r huashu-bookwriter ~/.claude/skills/

### md2book Skill（Markdown 转 PDF/HTML）
git clone https://github.com/ZeroxZhang/md2book
# 按 README 安装 pandoc + XeLaTeX 依赖

装好之后对 Claude Code 说"帮我写一本 XX 从入门到精通"或者"做个橙皮书"，Skill 就会自动触发。

这个事我用业余的时间做了3天，其实工作量还好，但我也有几个感受。

第一，创作者的技能是要被尊重的，所以，我这次只是致敬和学习，@花叔 如果有不当或者冒犯。我会随时下掉并公开致歉。

第二，逆向的过程中我最大的收获是：好的写作输出系统，是个非常庞大精密的体系，它太难一蹴而就了。风格一致性、质量把控、读者体验，每一条都有明确的执行标准和验收规范，我们要对行业专家保持敬畏。

第三，我知道这个 Skill 还原得还不完美。花叔的原始版本肯定有更多我没拆解到的细节。如果你用了之后有改进建议，欢迎在 GitHub 提 issue，或者直接评论。
