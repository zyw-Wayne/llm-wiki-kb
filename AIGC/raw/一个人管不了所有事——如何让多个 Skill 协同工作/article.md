# 一个人管不了所有事——如何让多个 Skill 协同工作

## 先说一个真实的困境
很多人用 OpenClaw 一段时间后，会遇到这样的问题：

Skill 越装越多，但感觉它们各管各的，彼此之间没有联系。

想做一件稍微复杂一点的事，比如「搜索今天的 AI 新闻，总结成简报，然后发给我」——你发现这需要三个 Skill 配合，但它们根本不知道彼此的存在。

这篇文章就来解决这件事：**如何设计让多个 Skill 协同工作的系统。**

## 一、为什么 Skill 不能什么都塞进去？
很多人第一反应是：那我把所有功能都写进一个 SKILL.md 不就行了？

不行。原因有三个：

**原因一：上下文成本**

OpenClaw 在每次对话时会把 Skill 的内容注入系统提示。一个 Skill 越大，占用的上下文越多，留给真正有用信息的空间就越少。

**原因二：触发混乱**

一个 Skill 承担的职责越多，description 就越难写精准。结果是要么触发太宽泛（什么都触发），要么触发不了（AI 不知道这个庞然大物是干什么的）。

**原因三：维护困难**

一个功能出问题，你要在一大堆代码里翻找。拆分成小 Skill 之后，哪里出问题改哪里，互不干扰。

**结论：一个 Skill，只做一件事。**

## 二、单一职责原则
借用软件工程里的概念——**每个 Skill 应该有且只有一个改变它的理由。**

具体到实践，判断一个 Skill 是否职责单一，用这个问题自测：

> ❝用一句话描述这个 Skill 是干什么的，这句话里有没有「和」这个字？
如果有——拆。

```
搜索新闻 和 生成摘要  →  拆成两个 Skill
整理文件 和 发送通知  →  拆成两个 Skill
翻译内容 和 保存结果  →  拆成两个 Skill
```

## 三、Skill 的三种协作模式
理解了单一职责，接下来的问题是：拆开之后怎么让它们配合？

### 模式一：串行管道（Pipeline）
最常见的模式。A 的输出作为 B 的输入，B 的输出作为 C 的输入。

```
用户请求
    ↓
[search-news]  搜索原始信息
    ↓
[summarize]    提炼摘要
    ↓
[format-brief] 格式化输出
    ↓
用户收到简报
```
**实现方式：** 在每个 Skill 的执行步骤末尾，明确说明「将结果传递给下一步骤」，或者在主 Skill 的步骤里显式调用其他 Skill 的能力。

**示例写法：**

```
## 执行步骤（daily-brief）
1. 调用搜索能力获取今日热点（最近 24 小时）
2. 对搜索结果执行摘要提炼，每条控制在 50 字以内
3. 按重要性排序
4. 套用简报模板格式输出
```
注意：不需要显式写「调用 search-news Skill」，AI 会根据能力描述自动整合。重要的是**把每一步的输入输出说清楚**。

### 模式二：并行聚合（Fan-out / Fan-in）
多个 Skill 同时处理不同维度的信息，最后汇总成一个结果。

```
用户请求竞品分析
    ↓
[search-product-A]  搜索竞品A信息
[search-product-B]  搜索竞品B信息   ← 并行执行
[search-market]     搜索市场数据
    ↓
[compare-report]    汇总对比报告
    ↓
用户收到分析
```
**适用场景：** 需要从多个来源收集信息再综合判断的任务，比如竞品分析、多维度评估、综合报告。

**实现方式：** 在聚合 Skill 的步骤里明确说明「分别处理以下几类信息，最后汇总」：

```
## 执行步骤（competitor-analysis）
1. 分别搜索以下信息：
   - 竞品A的产品定位和核心功能
   - 竞品B的产品定位和核心功能
   - 市场整体规模和增速
2. 按统一维度整理三组信息
3. 生成对比矩阵
4. 输出差异化建议
```

### 模式三：条件路由（Conditional Routing）
根据用户输入的不同，自动路由到不同的 Skill 处理。

```
用户输入
    ↓
[router-skill]  判断意图
    ↙         ↘
[skill-A]    [skill-B]
写作类请求    搜索类请求
```
**适用场景：** 你有一个统一的入口，但背后需要根据不同情况走不同处理路径。

**实现方式：** 写一个「路由 Skill」，它的唯一职责是判断意图并引导处理方向：

```
---
name: smart-router
description: |
  当用户的请求涉及多种可能的处理方式时，作为第一步判断路由。
  触发词：帮我、我想要、能不能、请你。
  注意：仅在其他 Skill 均未匹配时激活。
---

## 执行步骤
1. 分析用户意图，判断属于以下哪类：
   - 信息获取类 → 使用搜索相关能力
   - 内容创作类 → 使用写作相关能力
   - 文件操作类 → 使用文件相关能力
   - 其他 → 直接响应
2. 按判断结果执行对应处理
3. 不确定时，向用户确认意图
```

## 四、避免 Skill 之间的「内耗」
Skill 协作最容易出现的问题不是不配合，而是**互相抢地盘**——两个 Skill 的 description 描述了相似的场景，AI 不知道该用哪个，结果两个都不用，或者用错。

**检查方法：** 把所有 Skill 的 description 列在一张表里，横向对比触发场景有没有重叠。

| Skill | 触发场景 |
| --- | --- |
| search-news | 搜索最新资讯、热点动态 |
| trend-scout | 询问行业趋势、未来方向 |
| daily-brief | 生成今日综合简报 |

上面这三个看起来都跟「新闻/资讯」有关，但触发场景其实是不同的：search-news 是原始搜索，trend-scout 是趋势分析，daily-brief 是综合简报。只要 description 写清楚了区别，AI 就能正确路由。

**如果你发现两个 Skill 触发场景真的完全重叠——合并它们，不要硬维护两个。**

## 五、用 MEMORY.md 传递跨 Skill 的状态
OpenClaw 的 Skill 之间默认是无状态的——A Skill 处理完的结果，B Skill 不知道。

但你可以用 `MEMORY.md` 或自定义文件作为「共享黑板」，让多个 Skill 读写同一份状态。

**示例场景：** 你有一个「项目追踪」系统，多个 Skill 都需要知道「当前在做哪个项目」。

在 `MEMORY.md` 里维护：

```
## 当前上下文
活跃项目：OpenClaw Skill 开发
当前阶段：调试触发问题
最近更新：2026-03-27
```
然后每个相关 Skill 在执行步骤里加一条：

```
1. 读取 MEMORY.md 中的「当前上下文」
2. 基于当前项目背景处理用户请求
...
最后一步. 如有必要，更新 MEMORY.md 中的相关字段
```
这样多个 Skill 就共享了同一份上下文，协作起来更流畅。

## 六、一个完整的协作示例
假设你要搭建一个「每日 AI 内参」系统，每天早上自动生成一份个性化简报。

**需要的 Skill 清单：**

```
daily-brief        主 Skill，负责编排整个流程
  └─ search-ai     搜索 AI 行业最新动态
  └─ search-tools  搜索新工具发布信息
  └─ summarize     对每条信息提炼摘要
  └─ format-brief  套用简报模板格式化输出
```
**daily-brief 的执行步骤：**

```
## 执行步骤
1. 获取今日日期
2. 搜索过去 24 小时内的 AI 行业动态（限 5 条）
3. 搜索过去 24 小时内的新 AI 工具发布（限 3 条）
4. 对每条结果提炼 30 字以内的摘要
5. 按「重要性 > 实用性 > 趣味性」排序
6. 套用以下格式输出完整简报：
   - 日期标题
   - 今日要闻（搜索结果）
   - 新工具速览
   - 一句话点评
7. 若某类搜索无结果，该部分标注「今日暂无」，不得补充虚假信息
```
整个系统里，每个 Skill 只做自己那一件事，daily-brief 负责编排，其他 Skill 负责执行。清晰、可维护、可单独调试。

## 总结

| 原则 | 具体做法 |
| --- | --- |
| 单一职责 | 一个 Skill 只做一件事，描述里没有「和」 |
| 串行管道 | 明确每步的输入输出，让结果自然流转 |
| 并行聚合 | 多路收集信息，最后汇总到一个出口 |
| 条件路由 | 写路由 Skill 判断意图，不要让 AI 猜 |
| 避免内耗 | description 触发场景互相排他，不重叠 |
| 共享状态 | 用 MEMORY.md 作为跨 Skill 的黑板 |

**单个 Skill 做对一件事，多个 Skill 协作做成一件大事。** 这才是 OpenClaw 真正的用法。
