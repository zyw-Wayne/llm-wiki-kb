# 流式架构与性能优化

## 定义

Agent Harness 的流式架构是一种以"增量、可中断、可组合"为原则设计的系统范式。流式处理不是优化手段，而是架构选择——它决定了从 QueryEngine 到 StreamingToolExecutor 到工具执行的每一个组件都必须支持逐步推进和实时响应。

## 核心观点

- **流式架构是基石，不是优化**：它决定了系统的每个组件必须按"增量、可中断、可组合"的原则设计
- **工具并发靠"安全分类"**：Read/Grep/Glob 可并行（只读），Bash/Edit/Write 串行排队（有副作用）
- **结果顺序必须严格保持**：模型推理依赖工具结果的顺序，乱序会导致模型误解
- **启动性能靠"与 import 链赛跑"**：把 I/O 操作与模块评估并行化，可节省近 60% 启动时间
- **工具输出的 token 成本往往高于模型输出**：一次大文件读取可产生 60K+ 缓存创建 token

## 详细内容

### QueryEngine：查询生命周期管理者

`QueryEngine` 是会话状态的唯一所有者，遵循"单一所有权"原则。一个会话对应一个实例，状态（messages、文件缓存、usage 等）在 turn 之间持久保持。

**为什么用类而非函数式 API？** 状态管理的复杂性。类将状态封装为实例属性，新增状态只需在构造函数中初始化，不破坏现有接口。销毁实例即销毁状态，避免全局状态的内存泄漏。

`submitMessage` 是一个 `AsyncGenerator`，调用者可以实时消费流式产生的消息，不需要等待整个查询完成。流式处理的价值：工具1在第1秒就开始执行，到第5秒时三个工具可能都已执行完毕；非流式模式则需先等5秒接收完整响应再串行执行。

### StreamingToolExecutor：并发控制

"流到即执行"——工具在流式到达时立即启动，不等所有工具调用块全部到达。

**并发安全分类：**

| 类别 | 工具 | 规则 |
|------|------|------|
| 并发安全 (safe) | Read、Grep、Glob、WebFetch | 纯读取，无副作用，可并行 |
| 非并发安全 (unsafe) | Bash、Edit、Write、NotebookEdit | 修改文件系统，串行独占执行 |

`canExecuteTool` 的核心逻辑：
- 没有工具在执行时，任何工具都可以启动
- 有工具在执行时，只有所有参与方都是 concurrency-safe 才允许并行
- 非并发安全工具必须独占执行

**默认值原则（fail-closed）**：当不确定时，选择 `isConcurrencySafe = false`。宁可多等一秒，也不冒险并行执行可能冲突的命令。

**结果缓冲与顺序输出**：工具可以并行执行，但结果的输出必须保持顺序（按添加顺序依次 yield）。Progress 消息是例外，会被立即 yield，不受顺序约束。

**兄弟工具错误级联**：只有 Bash 工具失败才触发级联取消（通过 `siblingAbortController`），因为 Bash 命令之间往往存在隐式依赖链。Read/WebFetch 等独立工具的失败不影响其他并行工具。

### 启动性能优化

系统通过三类策略缩短从命令行输入到首次响应的延迟：

**1. 并行预取（与 import 链赛跑）**

在入口文件最早阶段并行启动两个子进程，与后续约 65ms 的 import 链并行执行：
- `startMdmRawRead`：读取 MDM plist 配置（30ms）
- `startKeychainPrefetch`：两个 keychain 读取并行（32ms + 33ms → max 33ms）

收益：串行 160ms → 并行 65ms，节省 95ms（59%）

适用条件：I/O 密集型、结果不会立即使用、失败不阻塞主流程。

**2. 延迟加载**

- `messageSelector` 使用 `require()` 实现惰性加载（拉入 React/ink 等重量级依赖，仅首次需要时加载）
- 条件导入：通过特性开关门控，未启用功能通过 dead code elimination 从构建产物中消除

决策阈值：模块加载时间超过 5ms 且不是启动后前 3 秒必需的，就应该延迟加载。

**3. 惰性 Schema 评估**

工具的 inputSchema 通过 `lazySchema` 实现延迟初始化。60+ 个工具的 Zod schema 如果在启动时全部构建，总开销 6-18ms。`lazySchema` 利用 getter 的"一次触发，永久替换"特性解决此问题。

### Token 成本追踪

两层函数设计：
- **`updateUsage`**：处理单条流式事件，使用 `> 0` 守卫区分"未返回"和"真实为零"（单调递增保证）
- **`accumulateUsage`**：跨消息累加总量

> 反模式：不要用 `|| 0` 作为默认值。`|| 0` 会将 `0`、`undefined`、`null` 都视为 falsy，导致真正的零值被错误替换。

QueryEngine 按消息累加；forked agent 按事件累加（更频繁，防止崩溃时丢失数据）。

**Token 预算分级响应：**

| 预算使用率 | 建议动作 |
|-----------|---------|
| < 50% | 正常运行 |
| 50–80% | 轻量级提醒 |
| 80–95% | 明确警告，建议压缩上下文 |
| > 95% | 强烈警告，限制新工具调用 |
| ≥ 100% | 硬限制，停止对话循环 |

### 提示缓存优化

缓存键由五个维度构成：system prompt、tools、model、messages prefix、thinking config。任何维度的偏差都会导致缓存断裂。

**Fork 模式的字节级一致性**：fork 请求必须与父请求共享完全相同的消息前缀。`contentReplacementState` 的克隆保证了 wire 格式的字节级一致性（逻辑等价 ≠ 字节等价，缓存键基于字节精确匹配）。

**缓存命中的经济影响**（Claude 3.5 Sonnet）：
- 普通输入：$3.00/1M tokens
- 缓存创建：$3.75/1M tokens（+25%）
- 缓存读取：$0.30/1M tokens（-90%）

典型会话缓存命中率 80% 时，成本可节省 72%。

**常见缓存断裂原因：**
- 时间戳注入到 system prompt（每次请求不同）
- 动态工具列表顺序随机（应排序后注入）
- Fork 使用了不同的 `maxOutputTokens`（影响 thinking config，进而使缓存失效）

**健康指标：**

| 指标 | 健康范围 |
|------|---------|
| 整体缓存命中率 | > 70% |
| Fork 缓存命中率 | > 85% |
| 缓存创建/读取比率 | < 0.3 |
| 单次请求缓存创建 token | < 50K |
