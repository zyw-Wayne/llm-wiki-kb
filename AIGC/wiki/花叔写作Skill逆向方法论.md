# 花叔写作 Skill 逆向方法论

## 定义
通过分析已出版的技术书籍（花叔的四本橙皮书）和开源 Skills 仓库，逆向推导出其写作 Skill 的底层框架和设计原则。这是一种"从产物倒推工具"的方法论。

## 核心观点
- 好的技术写作输出系统是庞大精密的体系，难以一蹴而就（来源：[花叔不公开的写作 Skill，我逆向出来了](../raw/花叔不公开的写作%20Skill，我逆向出来了/article.md)）
- 风格一致性不靠天赋，是靠规则（来源同上）
- 输出不能"差不多就行"，必须"通过了才放行"（来源同上）

## 逆向发现的五层架构

### 第一层：渐进式信任建立
- 每章开头有时间线锚点，如"用了 3 个月 Cursor 之后切换到 Claude Code"
- 规则：每章前 2-3 段必须有具体时间线锚点或场景还原
- 没体验就诚实标注，不编造
- **目的**：建立可信度，让读者知道作者真正用过这个东西

### 第二层：结构化的知识传递路径
- 统一递进逻辑：Part 1 从零到第一次跑通 → Part 2 深入核心能力 → Part 3 多场景实战
- 每章解决一个具体问题，读者读完一章能做一件事
- 章节之间有向前桥接，不让读者迷路

### 第三层：风格 DNA
花叔的写作规则（技术写作精确性要求）：
- **短句规则**：单句不超过 25 字，用句号断句
- **第一人称**：高频出现，每千字至少 3 次
- **具体数字**：不用模糊量词
- **禁用词表**：不用"综上所述"、"值得注意的是"、"接下来我们将"
- **体验原则**：有体验就断言，没体验就诚实标注
- **替代词对照表**："进行操作"→"点击"或"输入"，"实现功能"→"做到"或"搞定"

> 这种精确性不是"写得好看"，是写得精确。

### 第四层：TDD 式质量铁律
- **12 项章节 QC**：结构（4项）、风格（4项）、内容（4项）
- **10 项全书 QC**：整体一致性检查
- **Rationalization Table**：专门对付 Agent 找借口跳过规则的机制
  - 例：Agent 说"开头不重要，内容才是关键" → 表里反驳："花叔风格的核心就是开头。没有锚点的章节不可信。"
- 不通过就重写，不妥协

### 第五层：Agent Protocol
- 写作前必须先研究确认事实
- 检查版本信息、API 是否存在、社区反馈
- 不凭训练数据编造内容
- **目的**：保证内容不是训练数据的幻觉，是经过验证的事实

## 架构设计：Skill 拆分原则
- **单一职责**：创作 Skill 只管内容生成，格式转换单独做成 md2book Skill
- **复杂度控制**：创作 Skill 已有 8 个参考文件、3 个脚本、2 个模板、2 套测试，不宜再塞入导出逻辑
- **输出物**：创作 Skill 生成 Markdown → md2book 转 PDF/HTML

## 实践验证
- **DeerFlow 橙皮书**：用逆向 Skill 写的完整书籍，5 个 Part，10+ 节
- **还原度评估**：80% 以上，结构完全还原，风格细节有几处差一点
- **实验对象**：字节 DeerFlow 2.0（2026 年 2 月发布，GitHub Trending 第一）
