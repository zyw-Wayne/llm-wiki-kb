# Skill 测试与质量保障

## 定义

一套系统化的 Skill 质量保障方法论：从触发测试集设计、断言式输出评估到迭代优化闭环，将 Skill 从"能用"打磨到"生产可用"。核心可归结为三个原则。

## 三原则总览

| 原则 | 核心主张 | 对应质量维度 |
|------|---------|-------------|
| **Description 决定触发** | 不被正确调用，一切白搭 | 触发准确率（When） |
| **正文决定质量** | 触发后输出好不好，取决于 SKILL.md 正文 | 输出质量（What） |
| **测试决定可靠性** | 没有测试集的 Skill 是不可信的 | 持续迭代闭环 |

三者有严格的前后依赖：触发准确是前提 → 正文质量是基础 → 测试迭代是保障。

---

## 原则一：Description 决定触发

两个核心指标：

- **正例触发率**：该触发时被激活了吗？（避免漏触发 / False Negative）
- **负例误触率**：不该触发时保持沉默了吗？（避免误触发 / False Positive）

> **漏触发比误触发危害更大**，优先解决漏触发。用户说了相关的话却没反应，比偶尔多触发一次伤害更大。

### 构建触发测试集

测试的关键是构建一个包含正反例的问题清单，不需要写代码。

**测试用例结构**（JSON 格式）：

```json
{
  "query": "帮我把这个 PDF 的第 3-5 页单独提取出来",
  "should_trigger": true,
  "reason": "明确的 PDF 分割操作需求"
}
```

**正例 vs 负例设计**：

| 测试类型 | 目的 | 示例（用户输入） | 预期 |
|---------|------|----------------|------|
| **正例** | 确保被正确唤醒，覆盖同义词和不同表述 | "这个 PDF 扫描件里有文字，帮我提取出来" | `true` |
| | | "我有三个 PDF 要合到一起" | `true` |
| **负例** | 避免在无关场景下"捣乱" | "用 Python 写一个读取 PDF 的函数" | `false` |
| | | "PDF 和 Word 格式哪个更通用？" | `false` |

正负例比例约 **2:1**（正例覆盖面更宽）。

**测试规模递进**：

| 阶段 | 正例 | 负例 | 目的 |
|------|------|------|------|
| 初始验证 | 8-10 | 4-6 | 快速发现明显问题 |
| 优化阶段 | 15-20 | 10-15 | 细化触发边界 |
| 发布前 | 30+ | 20+ | 生产质量保证 |

### Description 优化四步法

当触发率持续不理想时的系统诊断方法：

1. 列出所有**漏触发**用例，找共性（未提到的同义词？未覆盖的格式别名？）
2. 分析**误触发**用例（太宽泛？某个词有歧义？）
3. 重写 description：补同义词（"表格"/"电子表格"/"spreadsheet"）、格式别名（`.xls`/`.xlsx`/`.csv`）、明确排除说明
4. 重跑测试集，对比触发率变化

## 原则二：正文决定质量

一旦正确触发，输出质量由 SKILL.md 正文决定。通过定义**断言（Assertions）**进行评估：

### 三种断言类型

| 类型 | 用途 | 示例 |
|------|------|------|
| **contains** | 输出必须包含某内容 | `"提取成功"` 提示、`.txt` 输出文件 |
| **not_contains** | 输出不应出现某内容 | `"抱歉，我无法处理"` |
| **format** | 输出必须符合格式 | `markdown_table`、特定报告结构 |

### 定性评估

断言无法覆盖的维度（注释准确性、排版专业度、错误提示友好度）用 **1-5 分制**人工评估，每轮迭代对比分数变化。

### 典型问题三大类

| 问题类型 | 表现 | 修复方向 |
|---------|------|---------|
| description 覆盖不足 | 用户说 `.xls` 但 description 只写了 `.xlsx` | 补充同义词和格式别名 |
| 正文步骤不完整 | 触发了但说"不支持此操作" | 补充操作路由和脚本 |
| 脚本错误处理缺失 | 异常输入导致技术错误直接展示给用户 | 加 try/except + SKILL.md 错误处理章节 |

## 原则三：测试决定可靠性

测试不是一次性的，而是持续迭代的闭环：

```
写初稿 → 运行测试 → 人工审查 → 分析问题 → 修改 Skill → （循环）
```

### 迭代策略

**第一轮**：发现大问题——优先解决漏触发（扩充 description），再处理误触发（收窄边界）

**后续轮次**：每次修改后重跑全量测试集，确保不引入回归。逐步扩大测试规模，覆盖边界情况（空文件、超大文件、格式异常）和与其他 Skill 场景重叠的案例。

### 在 Claude.ai 中的模拟测试法

无需自动化框架，用对话模拟即可：

```
你：请按照以下 SKILL.md 的指导，回答用户的问题：
[粘贴 SKILL.md 内容]

用户问题：[测试用例]
```

观察行为是否符合预期，发现问题后告诉 Claude 具体哪里不对，让它修改 Skill。

## 发布前质量检查清单（10 项）

1. ✅ description 覆盖主要触发场景（含同义词）
2. ✅ description 明确了不触发的场景
3. ✅ SKILL.md 正文 500 行以内
4. ✅ references 文档超过 300 行时有目录
5. ✅ 所有脚本有错误处理
6. ✅ SKILL.md 说明了所有脚本的用途和参数
7. ✅ 输出格式在 SKILL.md 中明确定义
8. ✅ 测试了至少 10 个正例和 5 个负例
9. ✅ 测试了主要边界情况
10. ✅ Skill 行为与 description 声明完全一致

## 实战案例：code-reviewer 三版演进

| 版本 | description | 正文 | 结果 | 问题诊断 |
|------|------------|------|------|---------|
| v1 | `代码审查工具。`（5 字） | 无 | 几乎不触发 | description 过短，模型无法判断何时该用 |
| v2 | `当用户需要代码审查、Code Review 时使用。`（15 字） | `帮用户审查代码，找出问题。` | 触发改善，输出不稳定 | 正文空泛，Agent 缺乏明确指导 |
| v3 | 含 5 种触发词 + 适用所有语言 + "**必须**使用此技能"推动性语句（60+ 字） | 5 维审查框架 + 报告模板 + 优先级规则 + 行号要求 | 触发准确、输出统一 | — |

**v3 的三个关键改进**：

1. **丰富 Description**：明确触发场景（"提交代码并要求...查找 Bug"）、适用范围（"所有编程语言"），增加推动性（"**必须**使用此技能"）
2. **结构化正文**：规定审查的**五大维度**（正确性 > 安全性 > 性能 > 可读性 > 可维护性）和优先级
3. **定义输出格式**：要求结构化报告（🔴严重/🟡建议/🟢优点），并包含行号和修复建议

此案例完美印证三原则：description 从模糊到精准（原则一）→ 正文从空泛到结构化（原则二）→ 通过测试驱动迭代（原则三）。
