# 代码执行范式 (Code Execution)

**分类**: 模式  
**标签**: Agent、MCP、代码执行、Token 优化

---

## 定义

**代码执行**是 Anthropic 提出的一种 AI Agent 新范式，核心思想是：

> **别再让模型直接调用工具了，让它写代码来调用工具**

该范式建立在模型上下文协议（MCP）之上，将 MCP 服务器呈现为代码 API，而不是直接的工具调用接口。

## 核心问题：Agent 的两大"隐形税"

### 1. 工具定义过载 (Tool Definition Overhead)

| 问题 | 描述 |
|------|------|
| 原因 | 将所有可用工具定义一次性加载到模型上下文 |
| 影响 | 连接数千个工具时，仅定义就消耗数十万 Token |
| 本质 | 模型还没开始工作，成本就已经产生 |

### 2. 中间结果消耗 (Intermediate Result Consumption)

| 问题 | 描述 |
|------|------|
| 原因 | 工作流中每一个中间结果都必须经过模型上下文 |
| 影响 | 大数据在处理过程中被模型重复处理多次 |
| 风险 | 可能直接超出上下文窗口限制 |

**示例**: 一份 5 万 Token 的会议纪要在传统流程中被模型处理了两次（读取时一次，写入 Salesforce 时一次）。

---

## 解决方案：代码执行工作流

### 文件树结构

```
servers/
├── google-drive/
│   ├── getDocument.ts
│   └── ... (other tools)
├── salesforce/
│   ├── updateRecord.ts
│   └── ... (other tools)
...
```

每个工具文件内部封装了对 MCP 工具的实际调用。

### 代码示例

```typescript
// 从 Google Docs 读取纪要并添加到 Salesforce
import * as gdrive from './servers/google-drive';
import * as salesforce from './servers/salesforce';

const transcript = (await gdrive.getDocument({ documentId: 'abc123' })).content;
await salesforce.updateRecord({
  objectType: 'SalesMeeting',
  recordId: '00Q5f000001abcXYZ',
  data: { Notes: transcript }
});
```

### 关键变化

| 特性 | 传统方式 | 代码执行方式 |
|------|----------|-------------|
| 工具加载 | 开局加载所有定义 | 按需浏览文件系统发现 |
| 数据流转 | 经过模型上下文两次 | 在代码变量中本地流转 |
| Token 消耗 | 15 万 | 2000 |
| 效率 | 基准 | **提升 98.7%** |

## 五大核心优势

### 1. 渐进式披露 (Progressive Disclosure)

模型像人类程序员一样工作：
- 通过探索文件系统（`ls ./servers/`）发现可用服务
- 使用 `search_tools` 工具按需学习工具用法

### 2. 上下文高效的工具结果

在代码执行环境中过滤、转换、聚合海量数据，只返回小规模结果给模型。

```typescript
const allRows = await gdrive.getSheet({ sheetId: 'abc123' });
const pendingOrders = allRows.filter(row => row["Status"] === 'pending');
console.log(`发现 ${pendingOrders.length} 个待处理订单`);
console.log(pendingOrders.slice(0, 5)); // 只记录前 5 个供模型审查
```

### 3. 更强大的控制流

用标准代码模式实现循环、条件、错误处理：

```typescript
let found = false;
while (!found) {
  const messages = await slack.getChannelHistory({ channel: 'C123456' });
  found = messages.some(m => m.text.includes('deployment complete'));
  if (!found) await new Promise(r => setTimeout(r, 5000));
}
```

比"调用工具 - 休眠 - 调用工具"的循环更高效。

### 4. 保护隐私的操作

中间数据保留在执行环境中，敏感数据自动令牌化：

- Agent 代码处理 `row.email` 和 `row.phone`
- 模型看到的是 `[EMAIL_1]` 和 `[PHONE_1]`
- 真实数据在环境中安全流转，不经过模型

### 5. 状态持久化与技能 (Skills)

**状态持久化**:
```typescript
await fs.writeFile('./workspace/leads.csv', csvData);
```

**技能积累**:
```typescript
// ./skills/save-sheet-as-csv.ts
export async function saveSheetAsCsv(sheetId: string) { ... }

// 在其他执行中复用
import { saveSheetAsCsv } from './skills/save-sheet-as-csv';
const csvPath = await saveSheetAsCsv('abc123');
```

## 权衡与挑战

### 收益

- Token 成本大幅降低（98.7%）
- 延迟缩短
- 工具组合能力提升

### 新挑战

- 需要安全的沙箱环境
- 资源限制机制
- 监控机制

## 代码执行范式在 Skill 框架中的落地

Skill 框架是代码执行范式的具体工程实现（来源：[[Skill框架源码分析]]）：

- `skill_run` 就是代码执行范式的核心工具——LLM 通过 shell 命令调用工具，而非直接 function calling
- 沙箱隔离（本地模式 + Docker 容器）解决"安全的沙箱环境"这一新挑战
- 增量暂存 + 符号链接 + 只读保护 = 生产级安全
- Token 节省效果一致：skill_run ~20 Token vs 8 个 function calling ~1600-4000 Token
