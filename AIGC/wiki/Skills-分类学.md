# Skills 分类学

## 定义

将生物分类学（界门纲目科属种）的层级思维应用于 Skill 组织管理——核心不是分得越细越好，而是找到最合适的颗粒度，让每个 Skill 在最顶层分类中占据一个不可替代的位置。

## 核心观点

Skill 的核心就两个词：**分类**和**触发**。Skill 怎么触发、能不能正确触发、触发以后能干什么，才是最核心的事。

## Skill 数量与准确率的关系

研究数据（论文实验）：

| Skill 数量 | 触发准确率 | 备注 |
|-----------|-----------|------|
| < 20 | > 90% | 几乎不会错 |
| > 30 | 明显下降 | 开始出现误触发 |
| ~200 | ~20% | 速度极慢，Token 消耗爆炸 |

作者实践：常年保持在 30 个以下。

## 颗粒度判断：生物分类学类比

生物分类是"界→门→纲→目→科→属→种"逐层穿透，不是直接跳到最细的"种"。同理：

- 封面图和 PPT 配图之间的差异 → **不值得**各自占独立 Skill，它们只是"图片生成"内部的变异
- 图片生成和服务器管理之间的差异 → **值得**各自占独立 Skill

作者实践案例：将 NanoBanana API 的多种生图需求（公众号封面、小红书封面、PPT 配图等）合并为一个"图片生成"Skill，内部按上下文做二次分支，同时覆盖通用生图需求。

## 三条判断标准

一个 Skill 值不值得独立存在：

1. **边界清晰** — 它对应的场景有没有明确的边界
2. **高频复现** — 它对应的场景是不是会高频复现
3. **不可归并** — 它能不能归属进已有的 Skill 里

以及奥卡姆剃刀原则：**如无必要，勿增实体**——用不到的 Skill 就别装。

## 分类系统设计

最重要的是设计自己的分类系统，厘清哪些该放 CLAUDE.md、哪些该做 Skill：

- **CLAUDE.md**：全局性规则、项目约定、通用行为约束
- **Skill**：有明确边界、高频复现、需要独立触发逻辑的场景

## 社区讨论补充

来源：原文评论区讨论，多位实践者的经验汇总。

### Skill 如同命名空间

将 Skill 类比为编程中的 namespace——一个 Skill 是一个命名空间，内部按上下文做细分路由。这与作者"图片生成"Skill 内部分支的做法一致，是分类学层级思维在工程实践中的具体实现。

### 按领域分类优于按功能分类

实践者经验：按**领域**（图片生成、服务器管理、数据分析）而非**功能**（调用 API、读写文件、执行命令）组织 Skill，触发准确率更高。因为用户意图通常是领域导向的，功能只是实现手段。

### 精简带来的收益

有实践者分享从 60+ Skills 精简到 20 以内后：
- 触发准确率明显提升
- 响应速度加快（减少 Skill 匹配的 Token 消耗）
- 维护负担大幅降低

### CLAUDE.md 与 Skill 边界的进一步共识

评论区讨论进一步明确了分工：
- **CLAUDE.md**：全局行为约束、代码风格、安全规则等"始终生效"的规则
- **Skill**：领域特定工作流，有明确触发条件，"按需激活"

判断依据：如果一条规则在所有场景下都适用 → CLAUDE.md；如果只在特定场景下需要 → Skill。

### Skill 调用链

Skill 之间可以互相调用形成组合链，但需控制层级深度，避免调用链过深导致上下文膨胀和调试困难。与 [[Skills-多Skill协作]] 中的管道/路由模式呼应。
