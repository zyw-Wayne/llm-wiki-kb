# Skills 设计哲学

## 定义

Skills 的形成逻辑与设计哲学，从"中台思维"推导出 Skills 的本质——**公共的 Prompt 就是 Skills**，以及"恰好而非更多"的渐进式加载理念。

## 核心推导：从"中台"到 Skills

### 1. 复用的力量

借鉴软件架构中的"中台"理念：把相同功能抽离出来，减少重复建设。

大胆抽象：**软件开发的整个过程 = 所有 Prompt 的总和**。每个人在 VibeCoding 中都有高频"口头禅"（重复指令），每次都打字太慢。把这些"口头禅"提前封装好，用的时候一个快捷键调用——这就是 Skills。

> 公共的 Prompt 就是 Skills！

### 2. 按流程分类

软件开发流程是固定的：设计 → 开发 → 测试 → 运维。Skills 可以按此框架分类，让成千上万个 Skills 有了归属，好的经验和方法可以覆盖软件工程全周期。

## 设计哲学：恰好而非更多

> **不是让 AI 知道更多，而是让 AI 在恰当的时间知道恰当的事**

- AI 的上下文窗口是有限的"桌面"，把所有 Skills 信息堆上去，AI 会胡言乱语、幻觉增多
- 解决方案：**渐进式加载（渐进式披露）**
  - 输入 Prompt 时只带最基本的元信息（name + description）
  - AI 按语义匹配，匹配到才加载实际内容
  - 匹配不上的不占上下文空间

这与 [[Skill定义规范]] 中"渐进式信息披露"的三层架构一致。

## 经验为什么能沉淀了？

- AI 时代，人类语言已成为一门全新的"编程语言"
- 任何以文字承载的领域知识，AI 都能理解和运用
- 个人经验以 Skills 形式低成本在团队内共享
- Skills 数量达到一定规模 → AI 技能商店 → 复杂问题通过基础 Skills 组合解决

## 面向 AI 的系统设计

AI 时代，系统设计将从"面向人设计"转向"面向 AI 设计"：
- 一份数据需要两个版本：给人好理解的 + 给 AI 好理解的
- 这是 Skills 成功给我们的启发

## 开发方法论

### 归纳法 → 沉淀经验

从工作案例中总结提炼，固化为 Skills。本质是"人肉强化学习"——不断试错、迭代、提炼出可复用经验。

### 演绎法 → 应用经验

将沉淀的规律应用到特定场景解决问题，或反馈完善 Skills。

### 核心难点：向外洞察 + 向内觉察

- **向外洞察**：精准捕捉反复出现的问题。解决问题的能力会被 AI 取代，但发现有价值的问题是最难被替代的能力
- **向内觉察**：像禅修般观察自己解决问题的隐性过程，梳理清晰后沉淀为 AI 能理解的领域知识
- 利用 AI 自我总结：解决完问题后，加一句 prompt 让 Agent 自己观察和沉淀

## 五种设计模式与组合思想

Google 总结的五种 Skill 设计模式（来源：[[Skill-设计模式与打磨法]]）：**知识注入型**（references 注入专业知识）、**模板生成型**（assets 放输出模板）、**审查打分型**（references 放检查清单）、**反转采访型**（AI 先问后做）、**流水线型**（多阶段严格执行）。

五种模式不互斥——先用流水线定骨架，在关键环节嵌入其他模式，各司其职。这与本页「恰好而非更多」的理念一致：每个环节只加载该环节需要的知识。

## 汉堡店类比：Prompt vs MCP vs Skill

用"开汉堡店"来理解三者区别（来源：[万字干货！Agent Skills从入门到精通](../raw/万字干货！Agent Skills从入门到精通/article.md)）：

- **Prompt** = 顾客点单："做一个牛肉汉堡，不要洋葱！"（指令明确，但怎么做全看厨师心情）
- **MCP** = 厨房工具和食材：铲子、平底锅、牛肉饼、面包
- **Skill** = 秘制菜谱 + 员工守则："肉饼煎 3 分半钟，酱汁挤两圈半，做完清理灶台！"

Skills 规定了动作的先后顺序、质量底线和执行标准——AI 不再瞎猜，而是按部就班干活。

> 模型是大脑，Agent 是躯体，Skills 就是双手。

## Skills = 预制菜 vs Prompt = 现炒菜

对比理解（来源：同上）：

| 概念 | 类比 | 说明 |
|------|------|------|
| Skills | 预制菜 | 做法预设好，点菜名就出菜，又快又准还省 Token |
| Prompt | 现炒菜 | 每次从头交代，灵活但重复劳动 |
| Project | 食材 | 基础数据和上下文 |
| MCP | 物流和外卖系统 | 连接外部工具和数据的协议 |

## 人与 AI 关系的三个阶段

从 Skills 的视角看人机关系进化（来源：一文带你看懂 Skills）：

1. **用户**（用 Prompt）：一问一答，本质上是跟 AI 说话
2. **指挥官**（用 Skills）：一次性把能力体系交给 AI，AI 自主运转
3. **架构师**（用多个 Skills）：设计一个由 AI 组成的系统

> "你第一次有能力，把自己的能力，变成一个可以复制、可以叠加、可以在你睡觉的时候继续运转的系统。"

Skills 从三个角度分别加载：先扫目录 → 判断用得上才翻正文 → 需要脚本再加载。像手机 APP 一样不会同时打开 100 个。

> 现在只是学会怎么"问"AI，其实已经有点不够了。更重要的是学会怎么"教"AI。把你重复做的工作、团队里的隐性知识、那些"只有老员工才知道"的操作细节，封装成 .skill 文件。
