# 多智能体协作五种模式

## 定义

Anthropic 官方总结的五种多智能体协作架构模式，覆盖从最简单的生成器-验证器循环到完全去中心化的共享状态模式。核心原则：**从最简模式开始，观察瓶颈，逐步进化**，而非因架构听起来高大上就盲目选用。

## 五大模式速览

### 模式一：生成器与验证器（Generator-Validator）

最简单的多智能体模式，落地应用最多。

**机制**：生成器输出初稿 → 验证器按设定标准评估 → 通过则结束，不通过则附修改意见打回重写 → 循环直到通过或达到最大重试次数。

**最佳场景**：输出质量要求极高、评估标准可清晰量化的任务。如客服邮件回复、代码+测试生成、事实核查、合规性审查。

**致命局限**：
- 验证器天花板 = 你给的标准有多细（"检查内容好不好"等于走个过场）
- 生成与验证难度相当时，验证器抓不出毛病
- 容易死循环：必须设最大迭代次数 + 备用方案（转人工或带警告返回最佳版本）

### 模式二：编排器与子智能体（Orchestrator-Workers）

层级制架构。一个组长统筹规划，子智能体领任务干活。

**机制**：组长接收任务 → 决定分工 → 可自处理一部分，其余分发给子智能体 → 子智能体完成后返回结果 → 组长整合为最终成品。

**Claude Code 实现**：主智能体写代码改文件，需要搜索大代码库时后台唤起子智能体，各子智能体在独立上下文窗口中工作并返回精炼结论。

**最佳场景**：任务拆解清晰、子任务间几乎无互相依赖。如代码自动审查（安全/覆盖率/风格/架构四项独立检查）。

**致命局限**：
- 组长容易成为信息瓶颈（子智能体间的关键情报必须经组长中转）
- 排队干活拖慢速度（除非显式并行）

> 来源文章明确指出：对于绝大多数刚起步的项目，**无脑首选此模式**。它以极低协调成本扛住最广泛业务场景。

### 模式三：智能体团队（Agent Teams）

持久化工作组。队友长期在线，积累上下文和领域专长。

**机制**：协调员唤醒多个工人智能体作为独立进程 → 队友从共享任务池接单 → 多步骤自主完成 → 打卡表示完成。与编排器最大区别：**工人持久存在**，不会做完就销毁。

**最佳场景**：子任务完全独立且需要长期多步连贯操作。如巨型代码库迁移（每个队友独立接管一个服务，处理依赖/测试/部署）。

**致命局限**：
- 绝对独立性 = 死穴（队友间无法互通有无，产出可能打架）
- 判断何时全部完成困难（耗时差异大）
- 抢公共资源会冲突（需精密的任务隔离和冲突预防）

### 模式四：消息总线（Message Bus）

事件驱动的发布-订阅架构，适合智能体数量激增的场景。

**机制**：所有智能体只做两件事——发布消息和订阅消息。路由器负责精准投递。新智能体加入无需改底层代码，直接接收相关工作即可。

**最佳场景**：事件驱动的流水线、生态注定不断膨胀。如安全运营自动化（警报分拣 → 网络调查/身份分析 → 情报补充 → 响应协调）。

**致命局限**：
- 查 Bug 是灾难（连锁反应的日志记录要求极高）
- 路由准确率是命门（分错类或漏发消息 = 默默罢工，不报错）
- LLM 做路由灵活但有"抽风风险"

### 模式五：共享状态（Shared State）

完全去中心化，消灭所有中间商。所有智能体在持久化存储空间自由读写。

**机制**：智能体自主运作，盯着同一个数据库/文档库 → 有能处理的信息就拿去分析 → 有新发现直接写回 → 通常由初始问题启动，满足终止条件停机。

**最佳场景**：需要高度协作、实时互通重大发现的研究型任务。如复杂科研文献综述（论文/研报/专利/新闻四方发现直接汇聚，互相启发深挖方向）。

**核心优势**：彻底消除单点故障——任何智能体宕机，其他人照样读写。

**致命局限**：
- 重复造轮子、南辕北辙（没人管）
- "无限接话死循环"：A 写 B 补 A 看了又补 → 疯狂烧 Token 产出为零
- 必须靠强硬终止条件（时间预算/连续多轮无新花样/专职监督智能体强行关机）

## 架构选型对比

| 对比维度 | 模式A | 模式B | 核心区分点 |
|---------|-------|-------|-----------|
| 编排器 vs 团队 | 编排器 | 团队 | 工人需要保留记忆多久？短平快→编排器，长期连贯→团队 |
| 编排器 vs 消息总线 | 编排器 | 消息总线 | 流程能否预知？固定套路→编排器，变数极大→总线 |
| 团队 vs 共享状态 | 团队 | 共享状态 | 队友间需不需要抄作业？各扫门前雪→团队，实时互通→共享状态 |
| 消息总线 vs 共享状态 | 消息总线 | 共享状态 | 处理完就结案还是越攒越大？流水线→总线，知识积累→共享状态 |

## 速查决策表

| 需求特征 | 推荐模式 |
|---------|---------|
| 输出质量极严苛 + 评估标准明确 | 生成器与验证器 |
| 任务拆解清晰 + 子任务边界分明 | 编排器与子智能体 |
| 大规模并行 + 互不干扰的持久战 | 智能体团队 |
| 事件驱动 + 持续加挂新智能体 | 消息总线 |
| 高度协作 + 实时互通重大发现 | 共享状态 |
| 绝不允许单点故障 | 共享状态 |
| 刚起步、不确定选什么 | **编排器与子智能体**（默认推荐） |

## 核心设计原则

> 以上下文为中心的任务拆解原则：不是按智能体能干什么活来分工，而是按它们**需要什么上下文**来分工。

## 混合架构

生产系统往往是混合流派：
- 主干用编排器，局部切共享状态
- 消息总线做总分发，下游由团队模式工人攻坚
- 这些模式是乐高积木，非此即彼
