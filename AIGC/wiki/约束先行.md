# 约束先行

## 定义

「约束先行」是数字生命卡兹克提出的 Agent 使用核心心法——**在让 Agent 做任何事情之前，先把规范定好**。全局的规矩、项目的规矩、文件夹的规矩，从上往下穿透，一层套一层。没有规矩的地方，不动手。

这四个字的本质是：**你脑子里知道的东西，如果没有写进文档，对 AI 来说就是不存在的。**

## 为什么比 Prompt 技巧更重要

- Agent 的短期记忆会丢失，对话框一关全忘了
- 下次打开，它唯一能看到的就是你留下来的文档和记忆文件
- 文档里写了什么、是否足够清晰，直接决定了 Agent 每次醒来是「清醒的」还是「懵的」
- 很多 Agent 越用越蠢，根本原因是规范和记忆体系混乱

> 约束先行解决的不是「怎么写更好的 Prompt」，而是「怎么让 AI 持续知道你是谁、要什么、怎么做」。

## 四层约束体系（CLAUDE.md 分层架构）

以 Claude Code 为例，约束从上往下层层穿透：

```
┌─────────────────────────────────┐
│  全局 CLAUDE.md（用户目录下）      │  ← 最高指令和原则：你是谁、做事原则、协作方式
├─────────────────────────────────┤
│  项目级 CLAUDE.md（项目根目录）     │  ← 项目宪法：目录结构、命名规范、文件组织
├─────────────────────────────────┤
│  规范文档/设计文档/架构说明         │  ← 领域规范：具体工作流和技术决策
├─────────────────────────────────┤
│  记忆文件（Auto Memory 等）        │  ← AI 自己的笔记和历史对话记录
└─────────────────────────────────┘
```

**类比公司治理**：制度在最上面，部门规范在中间，具体操作流程在最下面。CEO 不可能每天挨个盯着员工干活，靠的是制度穿透下去。

**类比模拟经营游戏**：前期最重要的不是赶紧建建建，而是先把路网规划好。路网一旦规划歪了，后面再怎么优化都没招，只能铲了重来。全局 CLAUDE.md 是城市主干道，项目 CLAUDE.md 是片区支路，主干道规划好了，支路自然就顺了。

## 全局 CLAUDE.md 的内容设计

作者迭代了多个版本后的最终形态，每个板块本质上都是一种**约束**：

| 板块 | 约束维度 | 作用 |
|------|---------|------|
| 关于我 | 身份约束 | AI 知道在跟谁协作 |
| 第一性原理 | 思考约束 | 不因惯例照搬，回到问题本身 |
| 反谄媚 | 沟通约束 | 不拍马屁，给真实判断 |
| 约束先行 | 行为约束 | 新项目先建规范，没规范不动手 |
| 交互设计原则 | 质量约束 | 用户体验是最高准则 |
| 工作方式 | 交互约束 | 中文、结论先行、不问废话 |
| 开发习惯 | 工程约束 | 改完跑验证、不注释掉报错 |
| Git 与部署 | 部署约束 | commit 用英文、push 等我说 |

### 约束先行的具体条文

```
## 约束先行
无论开发项目还是知识管理项目，第一步永远是建规则：
新项目先写 CLAUDE.md，新目录先定结构约定
（什么放哪、怎么命名、何时清理）。
没有规范的工作空间不动手。

已有规范的项目，严格遵守其 CLAUDE.md 中的约定。
需要调整规范时先改文档、再改实践，不要反过来。
```

关键点：「先改文档、再改实践」——规则不是死的，但改规则也要走规则的路。否则 Agent 为了赶进度绕过约定，事后想补文档都不知道从哪补起。

## 交互设计原则（非程序员视角的补充）

作者作为用户体验设计师，在全局 CLAUDE.md 中加入的五条交互设计原则：

1. **为目标设计，不为功能设计**：先问「用户要完成什么」，再决定怎么实现
2. **不要让用户思考**：交互应该不言自明，需要说明书才能用就是失败
3. **系统承担复杂性**：能自动化的不手动，能推断的不让用户填
4. **渐进式展示**：先给核心，细节按需展开
5. **反馈引导行动**：不要只报告问题，要引导下一步

> 写进去之后 Claude 做出来的东西确实不一样了。

## 与相关概念的关系

### vs Harness Engineering

「约束先行」是 Harness Engineering 中 **Guides（引导控制）** 的通俗表达：

| 维度 | Harness Engineering | 约束先行 |
|------|-------------------|---------|
| 范畴 | 完整工程体系（引导+检测+反馈） | 聚焦于引导层（事前约束） |
| 受众 | 工程师、架构师 | 非技术用户、设计师、普通开发者 |
| 落地 | AGENTS.md + Linter + CI + 多 Agent | CLAUDE.md 分层体系 |
| 核心思想一致 | 文档是活的反馈循环 | 没写进文档的就不存在 |

约束先行可以理解为 Harness Engineering 的**最小可行实践**——你不需要 16 个 Agent 并行、不需要自定义 Linter，只需要一套写得好的 CLAUDE.md 分层体系。

### vs AI 编程行为准则（Karpathy 四原则）

- Karpathy 四原则是**具体的 Agent 行为规则**（Think Before Coding、Simplicity First 等）
- 约束先行是**建立规则的方法论**（怎么让这些规则持续生效、层层穿透）
- 两者互补：约束先行搭框架，Karpathy 四原则填内容

### vs Agent 配置系统

Claude Code 的六层配置源（pluginSettings → policySettings）是约束先行的**工程实现**。用户写的 CLAUDE.md 通过 projectSettings 或 userSettings 生效，配置系统的信任半径递减原则保证了约束的安全性。

## 实战启示

1. **花一小时写好 CLAUDE.md，能省无数小时返工**——这不是修辞，是真实的效率对比
2. **知识管理类工作同样需要约束**——不只是开发项目，画图、做研究报告、创建 skills 都需要规范
3. **约束要从「你是谁」写起**——身份、价值观、工作方式，这些是 AI 理解你的基础
4. **不要等乱了再收拾**——预先建立规范的成本远低于事后重构
5. **类比任何需要秩序的事情**：做交互设计是给用户行为建约束，玩模拟经营是给虚拟城市建约束，开公司是给业务和人建约束，跟 Agent 协作还是在建约束
