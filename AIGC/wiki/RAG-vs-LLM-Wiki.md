# RAG vs LLM Wiki

**分类**: 决策  
**标签**: RAG、知识库、技术对比

---

## 核心对比

| 维度 | RAG | LLM Wiki |
|------|-----|----------|
| 知识存储 | 向量数据库 | Markdown 文件 |
| 检索方式 | Embedding 相似度 | 关键词 + 索引 |
| 知识积累 | ❌ 无 | ✅ 有 |
| 技术复杂度 | 高 | 低 |
| 适用规模 | 十万级 + | 几百到几万条 |

## RAG 的问题

**没有知识积累**:
- 今天回答复杂问题，花很多 token 分析文档
- 明天问相关问题，又要重新分析一遍
- 上次的思考过程没有沉淀下来

> 就像每次做数学题都要重新推导公式，而不是套用已证明的定理

## LLM Wiki 的优势

**编译阶段**: LLM 已读完、理解完、做好笔记  
**查询阶段**: 翻笔记，不是翻原书  
**积累效应**: 好回答存回 wiki，知识库越来越聪明

## 缓存层级类比

LLM Wiki 三层结构暗合 CPU 缓存设计:

| 层级 | 内容 | 访问特点 |
|------|------|----------|
| L1 | CLAUDE.md | 最小最快，每次必读 |
| L2 | index.md | 中等大小，先查索引 |
| L3 | wiki 页面 | 容量最大，必要时读 |

访问延迟：CLAUDE.md（几个 token）< index.md（几百 token）< wiki 页面（几千 token）

## 选择建议

**选择 RAG 如果**:
- 文档量在十万级以上
- 需要语义相似度检索
- 已有向量数据库基础设施

**选择 LLM Wiki 如果**:
- 文档量在几百到几万条
- 希望轻量级方案
- 重视知识积累效应
