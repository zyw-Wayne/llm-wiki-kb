# 上下文管理策略

## 定义

在 LLM 有限的上下文窗口中，平衡"记住足够多"和"别超标"的管理方法，常见策略包括滑动窗口、摘要压缩、RAG 检索增强。

## 为什么需要上下文管理

| 问题 | 说明 |
|------|------|
| Token 限制 | 每个模型有上下文窗口上限（如 128K tokens），超出就截断 |
| 成本 | Token 数量直接决定 API 费用，历史越长越贵 |
| 速度 | 输入 token 越多，模型响应延迟越高 |

## 三大策略

### 策略 1：滑动窗口

只保留最近 N 轮对话，丢弃更早的消息。

- 优点：实现简单，效果直接
- 缺点：早期信息完全丢失

### 策略 2：摘要压缩

让 LLM 把旧的对话历史压缩成一段摘要。

```
原始历史（5000 tokens）→ 压缩后（200 tokens）
"用户先查了深圳天气（25°C 晴），然后查了今日科技新闻（共 3 条），
 接着要求将新闻翻译成英文。"
```

- 优点：保留关键信息，压缩比高
- 缺点：压缩本身消耗 token，且可能丢失细节

### 策略 3：RAG 检索增强

不把所有信息塞进上下文，而是存到外部知识库，需要时按需检索。

- 优点：上下文永远保持精简，可以"记住"无限量的历史
- 缺点：需要额外的向量数据库基础设施

## 策略组合

实际项目中通常组合使用：

```
┌─────────────────────────────────────────────┐
│ 1. 系统提示词                    (始终保留)   │
│ 2. 压缩摘要（旧历史）            (摘要压缩)    │
│ 3. 检索到的相关知识片段           (RAG)       │
│ 4. 最近 N 轮对话                 (滑动窗口)   │
│ 5. 当前用户输入                  (始终保留)   │
└─────────────────────────────────────────────┘
```

> 上下文管理没有银弹，需要按场景组合不同策略。

## Claude Code 对话循环内的七步预处理管线（来源：raw/御舆：解码 Agent Harness/article.md，Ch2）

在每轮 API 调用前，Claude Code 对话循环执行七步预处理，按轻到重顺序排列（先尝试最小代价方案）：

1. **工具结果预算**：对过大的工具结果截断或持久化到磁盘（类似"分页"机制）
2. **Snip 压缩**：直接截断过长的历史消息内容（最粗暴，先用）
3. **Microcompact**：缓存友好的轻量压缩，尽量复用已缓存 token，避免缓存全面失效
4. **Context Collapse**：细粒度折叠——将连续的确认性消息折叠为紧凑视图，信息不丢
5. **系统提示组装**：将基础系统提示与动态上下文合并（组装顺序稳定影响缓存命中率）
6. **Autocompact**：全量摘要，替换历史对话（所有轻量手段失效后的最后防线）
7. **Token 阻断检查**：超过硬性限制则直接返回错误，不再发起 API 调用（快速失败）

核心原则：**压缩手段从轻量到重量排列，每步都先尝试最小代价方案**——因为每步都会丢失一些信息，应该尽量延迟最激进压缩手段的使用。

## Claude Code 的工程级上下文管理（来源：[御舆：解码 Agent Harness](../raw/御舆：解码%20Agent%20Harness/article.md)）

Claude Code 第7章揭示了生产级 Agent 的上下文管理远比"三大策略"更精细：

**有效窗口约束**：有效窗口 = 模型窗口 - 预留输出令牌（最多 20,000），200K 模型实际可用约 183,616 令牌。

**四级警告状态**：
- 0-85%：安全区，正常工作
- 85-90%：WARNING（黄色警告）
- 90-95%：AUTOCOMPACT（自动压缩触发）
- 95-100%：BLOCKING（拒绝新请求）

**专业上下文策略组合**：

| 场景 | 推荐策略 |
|------|---------|
| 读取大量文件后 | Snip 清除已分析完的工具结果 |
| 长时间暂停后返回 | MicroCompact 自动清除过期缓存 |
| 连续实施多个功能 | 每功能完成后手动 /compact |
| 跨子系统重构 | 分阶段工作 + 记忆系统辅助 |

**主动优于被动**：70% 时主动 `/compact` 附带保留提示，比等 95% 自动触发效果更好——可以指导哪些信息最重要。

**记忆系统补充**：重要决策提前存入记忆文件，压缩后仍可通过读取记忆恢复关键上下文。
