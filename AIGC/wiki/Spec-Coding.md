# Spec Coding（规范驱动开发）

## 定义

**Spec Coding（Spec-Driven Development，SDD）**是一种 AI 辅助开发方法论，在编写代码之前先建立清晰、结构化的规范文档，然后让 Coding Agent 基于规范生成代码。

核心洞察：**从"代码是真理之源"转向"意图是真理之源"**。规范不再是静态文档，而是可执行的、活的产物（Living Artifacts），直接驱动代码生成和演进。

## Vibe Coding 的痛点（Spec 诞生的背景）

| 问题 | 描述 |
|------|------|
| 缺乏全局视野 | AI 只在局部上下文改动，难以理解对整体架构的影响 |
| 代码质量不稳定 | 异常处理、边界条件、空值检查等细节经常被忽略 |
| 技术债累积 | 缺少事先的实现 Plan，临时打补丁越来越多 |
| 可追溯性差 | 核心逻辑散落在对话记录，难以回溯"为什么这么做" |
| 上下文丢失 | 长对话中 LLM 天然容易丢失连贯性，靠"脑补"填补空白 |
| 团队协作困难 | 关键信息沉淀在"某个人和 AI 的对话"里，他人看不到设计意图 |

## 三种 AI Coding 模式对比（以 TRAE 为例）

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| **Vibe 模式** | 快速验证想法、搭原型、简单改动 | 给 prompt 直接跑，即兴头脑风暴 |
| **Plan 模式** | 范围清晰的功能迭代、模块化重构 | 先生成简短实施计划，确认后动手 |
| **Spec 模式** | 从零开始的复杂项目、跨多开发周期的任务 | 生成三阶段文档组，每阶段支持用户确认和 Refine |

**选择原则**：项目越简单越清晰 → 快速推进；项目越复杂越模糊 → 先把结构搭好。

## Spec 三文档结构

在 TRAE 中输入 `/spec` 后，Agent 生成三份文档：

1. **spec.md（项目范围）** — 全局概览：做什么、为什么、决策权衡、范围边界。整个项目的北极星文档。
2. **tasks.md（任务拆解）** — 子模块 → 子任务 → 具体执行步骤，Agent 实时勾选完成状态。
3. **checklist.md（验证清单）** — 完成后逐项核查：代码实现、功能验证、测试覆盖。

文档存储位置：`.trae/specs/<任务名>/`（Plan 模式存 `.trae/documents/`），可纳入版本控制。

## Spec vs. 上传文档

| 维度 | 上传文档 | /spec |
|------|---------|-------|
| 主动性 | 被动（Agent 参考） | 主动（Agent 执行） |
| 进度追踪 | 无 | 实时勾选 |
| 纠偏能力 | 无 | 随时把 Agent 拉回来 |
| 任务拆解 | 无 | 自动生成有顺序的执行步骤 |

> 两者不互斥：可以在生成 spec 时提供现有技术文档作为背景资料。

## Plan vs. Spec 选择指南

- **用 /spec**：从零构建复杂项目、范围存在较多不确定性、跨多个开发周期、Agent 走错代价大
- **用 /plan**：范围清晰边界明确的任务——加功能、部分重构、明确定义的 bug 修复

本质都是：**在动手写代码之前，先和 Agent 建立清晰共识**，Spec 是更长更完整的版本。

## 常见问题

- **Q: Spec 会让开发流程变慢吗？** — 简单任务不会触发 Spec 模式；复杂任务"花 10 分钟对齐方案"远比"花 2 小时写完再推倒重来"高效。
- **Q: 执行过程中可以修改文档吗？** — 可以。Agent 会在文档生成后暂停等待确认，可直接编辑或用自然语言告诉 Agent 修改意图。
- **Q: 文档存在哪里？** — `.trae/specs/` 目录，可纳入版本控制，作为项目知识资产长期保留。

## Spec 模式与 Claude Code Plan 模式的对比

Claude Code 内置了类似 Spec 模式的"Plan 模式"（详见 [[Plan模式与结构化工作流]]），两者的设计理念高度一致但实现层面有差异：

| 维度 | Spec 模式（TRAE） | Plan 模式（Claude Code） |
|------|-----------------|------------------------|
| 触发方式 | 用户主动输入 `/spec` | Agent 根据任务复杂度主动判断，或用户显式触发 |
| 产出物 | 三文档（spec.md / tasks.md / checklist.md） | 单一计划文件（slug-based .md） |
| 持久化 | `.trae/specs/<任务名>/` 目录，纳入版本控制 | `.claude/` 目录，三层恢复机制防丢失 |
| 验证机制 | checklist.md 手动核查 | 独立验证智能体自动对照实施结果 |
| 权限约束 | 无（文档层面的约束） | 系统级权限切换（规划阶段只读工具） |
| 协作 | 单用户工作流 | 支持 Teammate 模式，计划通过 mailbox 交 team lead 审批 |

**共同洞察**：两者都回答了同一个问题——"如何在 AI 执行复杂任务前建立清晰的意图对齐"。Spec 模式的"三文档结构"与 Plan 模式的"六步行为指令"本质上都是在构建从发散探索到收敛方案的认知框架。
