# Chain of Thought（思维链）提示词技巧

## 定义

Chain of Thought（CoT，思维链）是一种 Prompt 技巧：强制 AI 把推理过程显式地写出来，而不是直接跳到结论。核心原理是让模型"推出答案"而非"赌一个答案"。

触发方式极简：在任务描述后加一句「在开始写代码之前，先把你的思路梳理一遍，说说你打算怎么实现」。

## 核心观点

- AI 直接给结论时，给的是"统计上最可能正确的答案"，是在赌，不是在想（来源：[我在 Prompt 里加了四个字](../raw/我在 Prompt 里加了四个字，Claude Code 的输出质量提升了一倍/article.md)）
- 推理过程显式化后，模型不得不一步一步推，错误更容易在过程中被发现和修正
- Google 2022 年研究：CoT 在复杂推理任务上可将准确率提升 **20%–40%**
- 适用场景：需要判断、取舍、分析的任务；不适用于简单检索型任务

## CoT 在 Claude Code 中的四种用法

### 用法一：架构决策前的强制推演

技术选型时不要直接问"我该用 X 还是 Y"，而是给出结构化步骤：

```
我需要在 Next.js 14 里实现用户订阅状态的全局共享，有两个方案：
React Context 放在 layout.tsx 里，或者用 Zustand。

请按照这个顺序思考：
1. 先分析这个状态的特点：更新频率、使用场景、是否需要持久化
2. 基于分析，判断两个方案各自适合什么场景
3. 对照我的具体情况（...），给出明确的推荐
4. 说出你推荐这个方案的最关键理由（只要一条，最重要的那条）
```

**关键**：最后要求"只要一条"，迫使模型做取舍，而非把所有考虑因素都推给你。

### 用法二：复杂 Bug 的分步诊断

```
帮我系统性地排查这个问题，按以下步骤思考：
1. 首先列出可能的原因（至少 5 个）
2. 对每个原因，说明如何验证它
3. 按照"最可能出错"到"最不可能出错"排列
4. 从排名第一的原因开始，帮我写出检查代码
```

额外价值：CoT 让它列出的"可能原因清单"本身就有价值，能补上你自己排查时的盲区。

### 用法三：边界条件推演（出海/国际化场景）

涉及时区、货币、国际化逻辑时，先让它分析边界条件再实现：

```
在开始实现之前，先帮我分析这个功能的边界条件：
1. 时区问题：Unix timestamp 应基于 UTC 还是用户本地时区？
2. 重复发送问题：定时任务每小时跑一次，如何防止重复？
3. 取消订阅的用户：是否继续发剩余提醒？
4. 新用户：刚注册但周期恰好 7 天内，是否发"即将到期"？
把这四个问题都分析清楚，每个给出处理建议，然后再开始写实现代码
```

### 用法四：方案设计的自我质疑（最高阶用法）

设计完方案后，立刻让它从对立面质疑自己：

```
你刚才设计的这个方案，现在站在反对方的立场上，
列出这个方案的三个最大弱点，每个弱点说明在什么情况下会产生问题
```

**真实案例**：Stripe Webhook 幂等性设计用 Redis 记录 event_id，质疑阶段才暴露"Redis 重启或键过期会漏处理"的风险——设计阶段并未提及。

## 常见坑与解决方案

**坑一：推理过程太长，结论被淹没**

在 Prompt 末尾加格式约束：`思考过程用简短的要点形式呈现，最终结论单独一段加粗显示`。格式约束与 CoT 不冲突。

**坑二：分析到一半就开始写代码**

把分析和实现拆成两次对话：
- 第一次：`这次只分析，不写代码，等我确认方向之后再实现`
- 确认方向后，再开启第二次对话实现

额外好处：分析阶段可以纠正模型的判断，实现阶段方向经过确认，不会出现"写完了发现方向不对"。

## CoT vs ToT：线性推理 vs 树状探索

| 维度 | Chain of Thought | Tree of Thought |
|------|-----------------|-----------------|
| 推理路径 | 单条线性路径 | 多条路径并行探索 |
| 触发方式 | "让我们一步一步地思考" | "提出3种方法，列出优缺点，选最优展开" |
| 适用场景 | 逻辑链清晰的分析题 | 需要方案对比和取舍的决策题 |
| Token 成本 | 较低（单路径） | 较高（多路径+评估） |
| 选择建议 | 先用 CoT，不够再升级 ToT | ToT 是 CoT 的超集 |

## 适用判断

| 任务类型 | 是否需要 CoT |
|----------|-------------|
| 需要多步判断（架构设计、技术选型） | 需要 |
| Bug 排查（原因不明、涉及多个系统） | 需要 |
| 边界条件复杂（时区、货币、幂等性） | 需要 |
| 简单检索（查 API 用法、写样板代码） | 不需要 |
