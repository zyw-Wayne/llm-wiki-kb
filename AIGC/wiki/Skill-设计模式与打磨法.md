# Skill 设计模式与打磨法

## 定义

Google 总结的五种常见 Skill 设计模式，以及一套「评测驱动」的六步打磨方法论。核心思想：**不要猜 Skill 该写什么，让失败告诉你该写什么。**（来源：熊猫Jay 基于自身售前工时评估实战验证）

## 五种 Skill 设计模式

### 模式一：知识注入型（Tool Wrapper）

**解决**：AI 的通用知识不够精准，需要注入「只有你知道的」专业知识。

把专业经验打包成 `references/` 文件，AI 按你的标准做事。最简单的模式——一个 SKILL.md + references/ 即可，无编程门槛。

**典型场景**：写小红书文案但风格不对 → 放入历史爆款文案 references，AI 模仿调性。

### 模式二：模板生成型（Generator）

**解决**：AI 每次输出格式不一致，结构飘忽不定。

在 `assets/` 里放输出模板，AI 做填空题而非自由发挥。适用于日报、技术报告、产品需求文档等有固定格式要求的重复性产出。

**典型场景**：每天写日报 → 放入模板，AI 按模板填内容。

### 模式三：审查打分型（Reviewer）

**解决**：需要 AI 按固定标准检查、评估某东西。

把检查清单放 `references/`，SKILL.md 定义审查流程：加载清单 → 逐项检查 → 按严重程度分类 → 输出报告。换一份 checklist 就变成完全不同的审查工具。

**典型场景**：公众号文章质量检查 → checklist 含标题吸引力、开头钩子、排版规范等。

### 模式四：反转采访型（Inversion）

**解决**：AI 总在信息不足时急着动手，结果偏差很大。

翻转交互方向：AI 先当采访者，问清楚再动手。看起来多一轮对话，省了反复返工的时间。**在实际使用中被严重低估。**

**典型场景**：规划日本旅行 → AI 先问天数/预算/同行人/偏好，再出方案。

### 模式五：流水线型（Pipeline）

**解决**：任务复杂步骤多，AI 容易跳步或遗漏关键环节。

拆成严格阶段，每个阶段有明确输入、输出和检查点。上一步没通过验证不允许进入下一步。

**典型场景**：写公众号文章 → 选题确认 → 大纲 → 写初稿 → 自检 → 终稿。

### 选型决策树

| 核心需求 | 选哪个模式 |
|---------|-----------|
| AI 对某个工具/框架的用法不准 | 知识注入型 |
| 每次输出格式不统一 | 模板生成型 |
| 需要按固定标准做检查 | 审查打分型 |
| AI 总在信息不足时急着动手 | 反转采访型 |
| 任务步骤多且不能跳步 | 流水线型 |

> **这五种模式不是互斥的，完全可以组合。** 一个流水线型 Skill 可以在第一步用反转模式收集需求，在最后一步用审查模式做质量检查。

## 六步打磨法（评测驱动）

核心思路：**先定义「什么算做对了」，再写 Skill 去达标。**

### 第一步：建立基线，发现问题

先让 DeepSeek、Claude 直接做一遍任务，观察三个维度：
- 哪些环节表现不稳定（同样输入，结果时好时坏）？
- 哪些输入让 AI 理解歧义、走偏方向？
- AI 有没有在不该主动时「自作聪明」？

「五分钟测试」：如果裸跑 5 分钟内搞不定或结果时好时坏，才是 Skill 真正的切入点。

### 第二步：定义验收标准

把发现的问题翻译成 3-5 条具体的测试用例，每条有明确的「通过/失败」标准。没有测试约束的 Skill 本质上是在放大 AI 行为的不确定性。

### 第三步：写最小版本的 Skill

MVP 版本，只做三件事：
1. 写「什么时候不要用」—— 防止误触发
2. 定义最短成功路径 —— 最少步骤得到稳定输出
3. 保持单一职责 —— 一个 Skill 只解决一个问题

### 第四步：逐步扩展

最短路径跑通后，增加覆盖范围。**核心纪律：每增加一条新规则，都必须对应一个测试用例。** 知道什么时候该停——规则太多反而让 AI 行为不可预测。

### 第五步：加记忆和验收标准

- **记忆**：Skill 目录下放日志文件，每次执行完追加记录。只记录反复出现的模式和重要经验。
- **验收标准**：明确写出什么算成功、什么算失败。AI 可用这些标准做自检。

### 第六步：上线后持续校准

观察信号：
- AI 在未预期场景下误触发？→ description 需要收窄
- AI 执行时漏掉关键参考文件？→ SKILL.md 里的引用不够明确
- AI 反复读同一段内容？→ 那段内容应放到更醒目位置

发现问题后回到第二步补测试用例，形成持续闭环。

## 实战案例：售前工时评估 Skill

### 背景

客户催工时报价，AI 两次生成结果：157 人天 vs 71 人天，差一倍。拿 71 人天报价项目铁定亏。

### 六步对应

1. **基线**：需求分析太泛（不知道系统模块结构）、资源虚构（不知道团队有谁）、格式不统一、工时差距大
2. **验收标准**：四条——需求关联真实模块、人员输出真实姓名、工时有代码依据、报告结构固定七章
3. **MVP**：SKILL.md 锁死四步流水线 + 七章报告模板
4. **扩展**：补充系统模块地图（`module-structure.md`）、影响范围规则库（`impact_rules.md`）、员工能力矩阵（`employee-data.md`）；拆子 Skill 复用（`req-to-dev-doc`、`dev-doc-match-resource`）
5. **验收+脚本**：自检清单 + `generate_word_report.py` 脚本替代 AI 转格式
6. **校准**：发现需求太简略时 AI 过度脑补 → 加约束「信息不足先列待确认项，不要自行补全」

### 最终效果

| 维度 | 没有 Skill | 有 Skill |
|------|-----------|---------|
| 耗时 | 半天到几天 | 10-15 分钟 |
| 输出格式 | 每次不一样 | 七章固定，直接生成 Word |
| 人员配置 | 编造泛泛岗位名 | 按真实团队能力精准匹配 |
| 工时评估 | 拍脑袋 | 代码扫描 + 技能系数 |
| 交付质量 | 大量人工返工 | AI 自检后输出，微调即可 |

### 四种模式组合

- **流水线型**（主模式）：四步严格按顺序执行
- **知识注入型**：模块地图、规则库、能力矩阵作为 references
- **模板生成型**：七章报告模板写死
- **审查打分型**：AI 输出前按自检清单逐项检查

## 核心洞察

- Skill 填的不是技术的坑，是「理解」的坑
- 真正值钱的是 AI 永远猜不到的东西：模块地图、踩坑规则、团队能力矩阵
- 每次改动都在回应一次失败，这就是「评测驱动」的核心
- Claude Code 的执行过程是透明的（能看见每一步读了什么文件），适合打磨阶段；封装更深的工具分不清是 Skill 问题还是环境问题
