# Agent Harness 构建指南

## 定义

从零构建自定义 Agent Harness 的完整工程路线图，综合 Claude Code 源码中提炼的五大设计原则和六步核心组件实现方法。本指南聚焦于"为什么这样设计"而非"是什么"，帮助开发者在遇到新场景时做出一致的设计决策。

## 核心观点

- **循环优于递归**：`while(true)` + State 对象 + `continue` 比递归调用更易调试、状态恢复更自然、中止更可控
- **Fail-closed 默认值**：工具默认不安全、有副作用、需确认——显式声明安全性，而非默认允许
- **依赖注入隔离 I/O**：核心循环不知道自己运行在哪里，环境差异封装在适配器中
- **Agent Harness 的价值**：不是 API 调用封装，而是融合对话管理、工具编排、权限控制、上下文工程、记忆系统和可观测性的完整软件架构

## 详细内容

### 何时需要 Agent Harness

满足以下任一条件时应考虑 Agent Harness 模式：

1. 智能体需要根据中间结果决定下一步（自主循环）
2. 涉及文件系统、代码执行、网络请求等副作用操作
3. 对话可能跨越数十甚至数百轮交互
4. 需要精细的成本控制和 token 预算管理
5. 需要可观测性——追踪每一步的决策过程和资源消耗

决策经验法则：LLM 做"输入→输出"的转换（翻译、摘要、分类）→ 简单 API 调用；LLM 做"观察→思考→行动→再观察"的循环 → Agent Harness。

### 五大设计原则

**原则一：循环优于递归**

循环的三大优势：
- 状态恢复只需重赋值 `state` 变量（递归需回退整个调用栈）
- `AbortController` signal 检查在循环顶部是自然退出点
- 状态变化发生在固定代码位置，一个断点捕获所有转换

**原则二：Schema 驱动而非硬编码**

每个工具通过 Zod Schema 定义，`buildTool` 工厂函数自动填充默认行为。"单一真相源"——验证逻辑、权限检查、描述生成都从同一 Schema 派生，修改一处全局生效。

**原则三：渐进式权限**

四阶段短路管线：validateInput → checkPermissions → PreToolUse 钩子 → canUseTool（用户确认）。早期拒绝原则（Fail Fast）：输入不合法就不检查权限，权限拒绝就不弹确认对话框。

**原则四：流式优先**

所有数据通过 `AsyncGenerator` 的 `yield` 传递，消费端逐条处理，实时 UI 更新成为可能。

**原则五：可插拔扩展**

钩子系统在 20+ 个生命周期节点提供扩展点（SessionStart、PostToolUse、PreCompact、Stop 等），每个钩子是独立的 Shell 命令或 HTTP 端点，通过标准化 JSON 协议交互。

### 六步核心组件实现路线图

```
Step 1: 对话循环 (AsyncGenerator + while(true))
  ↓
Step 2: 工具系统 (buildTool 工厂函数)
  ↓
Step 3: 权限管线 (四阶段短路检查)
  ↓
Step 4: 上下文管理 (渐进式压缩策略)
  ↓
Step 5: 记忆系统 (提取/存储/注入)
  ↓
Step 6: 钩子系统 (Shell 命令扩展点)
  ↓
最小可运行 Harness
```

**Step 1：对话循环关键决策**
- `State` 对象模式：循环状态封装在单一变量，每个 `continue` 站点写入新的 State 对象
- 依赖注入：`deps` 对象封装所有 I/O 操作（模型调用、UUID 生成），使测试可注入模拟实现
- 生产必需补充：最大轮次限制、AbortController 中止机制、指数退避重试、Usage 追踪

**Step 2：工具系统（buildTool 工厂函数）**

Fail-closed 默认值集合：
```
isConcurrencySafe: () => false   // 默认不安全并发
isReadOnly: () => false          // 默认假设写入
isDestructive: () => false       // 默认假设有破坏性
checkPermissions: async () => ({ allowed: true })
```

工具显式覆盖以声明安全性。新工具在安全审查通过之前不会产生危险行为。

**Step 3：权限管线**

四阶段实现：
1. 工具自身验证（`validateInput`）
2. 工具自身权限检查（`checkPermissions`）
3. 用户配置规则匹配（`matchPermissionRules`）
4. 用户交互确认（仅交互模式）

生产增强：审计日志、动态规则更新、权限缓存（只读工具）、角色-Based 访问控制。

**Step 4：上下文管理（渐进式压缩）**

| 策略 | Token 节省率 | 信息损失 | 触发阈值 |
|------|------------|---------|---------|
| Snip（历史裁剪） | 20–40% | 高 | 70% 上限 |
| 微压缩 | 30–50% | 中 | — |
| 摘要压缩 | 70–90% | 低（有损） | 90% 上限 |

关键洞察：信息损失是相对的。选择策略时需要问"当前任务什么信息最重要"。

**Step 5：记忆系统三个核心挑战**
- 提取什么：用户偏好、项目约定、重要决策（高价值）vs 临时调试输出（低价值）
- 存储多久：根据时效性和引用频率决定，需衰减曲线
- 何时注入：根据当前对话相关性选择性注入，而非全量注入

**Step 6：钩子系统关键设计**
- 超时管理：10 秒超时，超时视为错误而非阻塞
- 错误容忍：钩子是"建议者"，失败不停止 Agent 工作
- 安全隔离：钩子在独立进程执行，防止恶意钩子影响主进程
- 退出码约定：`code 0` = success，`code 2` = blocking，其他 = error

### 架构教训

**循环依赖打破策略：**
- Lazy Require 模式：`require()` 运行时加载 + 特性开关
- 类型集中导出：类型定义与实现解耦，工具文件从集中位置导入类型

**工具集合不可变原则**：`Tools` 类型定义为 `readonly Tool[]`，工具集合创建后不可修改。好处：安全性保证、推理简化、缓存友好（哈希固定）。

**功能开关分层策略：**

| 开关类型 | 生命周期 | 适用场景 |
|---------|---------|---------|
| 编译时开关（feature()） | 构建时确定，永久 | 未完成的功能、A/B 测试 |
| 运行时配置开关（settings.json） | 会话级 | 用户偏好、环境差异 |
| 远程特性标志 | 实时，API 驱动 | 灰度发布、紧急禁用 |

**断路器模式：**

错误处理四层防御：
1. 流式错误拦截：错误包装为 AssistantMessage（`isApiErrorMessage` 标记）
2. 延迟显示：可恢复错误暂时扣留（withheld），尝试恢复后再决定是否展示
3. 断路器：连续失败计数达到阈值（3次）停止重试
4. 降级策略：切换到 fallback 模型，清理状态后重试

Claude Code 选择 3 次连续失败作为阈值：超过 3 次通常是系统性问题，重试无意义。

### 多环境适配

核心原则：**核心不知道自己运行在哪里**。如果核心代码出现 `if (isCLI) ... else if (isSDK)` 分支，说明依赖注入不彻底。

| 环境 | 权限交互 | UI 渲染 | 会话存储 |
|------|----------|---------|---------|
| CLI | 终端提示符 | Ink/React | 本地文件 |
| IDE (VS Code) | 弹窗 | WebView | 本地文件 |
| SDK | 回调函数 | 无 | 内存 |
| Server | 自动拒绝/允许 | 无 | 数据库 |

### 可观测性四层架构

**第一层（投入低，收益高）**：结构化日志，每个工具调用记录 `{tool, duration, input_size, output_size, status, query_chain_id}`。

**第二层（投入中，收益中）**：指标聚合，P50/P95/P99 延迟、错误率、token 消耗率、压缩频率。

**第三层（投入高，收益高）**：分布式追踪，通过唯一 `query_chain_id` 串联多 Agent 的所有操作。

**第四层（投入高，收益渐进）**：异常检测，基于历史数据识别异常模式（单轮工具调用数异常、token 突然飙升、错误率上升）。

### 安全威胁模型

Agent 系统的威胁与传统应用有本质不同：威胁来自 LLM 本身（可能因 prompt injection 执行非预期操作），而非仅外部攻击者。

| 威胁 | 防御措施 |
|------|---------|
| Prompt Injection | 工具结果隔离 + 权限检查 |
| 工具滥用 | 权限管线 + 用户确认 |
| 资源耗尽 | 预算上限 + 最大轮次限制 |
| 数据泄露 | 输入净化 + 工作区信任检查 |
| 供应链攻击 | 钩子白名单 + 沙箱执行 |

安全审计 Checklist：
- [ ] 每个工具有正确的 `isReadOnly`/`isDestructive`/`isConcurrencySafe` 标记
- [ ] 权限管线无绕过路径
- [ ] 预算上限在每轮循环中检查（非仅会话结束时）
- [ ] 子 Agent 权限不超过父 Agent 范围
- [ ] 所有权限决策有审计日志

### 未来演进方向

| 当前状态 | 未来演进 | 驱动力 |
|---------|---------|-------|
| 单体工具系统（单进程内模块集合） | 微服务工具系统（独立进程，标准化协议） | MCP 成熟 |
| 被动执行模式（等待用户指令） | 主动 Daemon 模式（监听环境变化主动行动） | 长期运行需求 |
| 单技能 Agent | 多技能编排器（动态组合技能模块） | 任务复杂化 |
| 隐式文本记忆（片段式） | 显式知识图谱（结构化记忆与推理） | 精确检索需求 |

**Daemon 模式的三大挑战**：状态持久化（跨进程生命周期）、错误恢复（崩溃后状态一致性）、资源管理（内存泄漏随时间累积）。
