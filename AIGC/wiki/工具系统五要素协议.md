# 工具系统五要素协议

## 定义

Claude Code 中每个工具必须遵循的统一类型契约 `Tool<Input, Output, Progress>`，通过五要素协议定义工具的完整行为规范。"接口即架构"——通过严格类型接口，工具系统的所有架构约束都被编译器强制执行。

## 核心观点

- 工具系统的可扩展性来自类型契约，添加新工具不需要修改编排引擎代码（来源：raw/御舆：解码 Agent Harness/article.md）
- `buildTool` 的默认值遵循"fail-closed"原则——安全性相关方法默认为 false，工具必须显式声明自己安全才能享受并发等优化
- 五要素协议覆盖工具调用的"生老病死"：从开始到进行中、成功、被拒、出错到并行展示

## 详细内容

### 五要素协议

**要素一：名称与别名**
- 唯一名称标识符 + 可选别名（向后兼容）
- 工具查找同时检查主名称和别名
- 工程原则：公开 API 中重命名是"只增不减"的操作

**要素二：Zod Schema**
- 双重职责：运行时验证（LLM 输出的参数校验）+ API 通信（生成 JSON Schema 发给 API）
- Schema 定义就是工具的"使用说明书"——模型看到的参数描述来自 `describe()` 调用
- "不要信任外部输入"原则的体现

**要素三：权限模型（三层）**
1. `validateInput`：数据合法性检查，与权限无关
2. `hasPermissionsToUseTool` + `checkPermissions`：工具特定权限逻辑
3. `isConcurrencySafe()` 等运行时属性：影响并发调度策略

**要素四：执行逻辑**
- `call` 方法接收：解析后的输入、工具使用上下文、权限检查函数、父消息引用、进度回调
- 返回：输出数据 + 可选的 `contextModifier`（上下文修改器，用于更新文件缓存等后续行为）

**要素五：UI 渲染（六个方法）**
| 渲染方法 | 触发时机 |
|---------|---------|
| `renderToolUseMessage` | 工具调用开始 |
| `renderToolUseProgressMessage` | 工具执行中进度 |
| `renderToolResultMessage` | 工具结果展示 |
| `renderToolUseRejectedMessage` | 权限被拒绝 |
| `renderToolUseErrorMessage` | 执行出错 |
| `renderGroupedToolUse` | 多个并行工具的分组展示 |

所有渲染方法返回 `React.ReactNode`，与 React 渲染管线深度集成。

### buildTool 工厂函数

`buildTool` 接收部分工具定义，自动填充安全默认值：
- 安全类方法默认 `false`（fail-closed）
- 工具必须显式声明 `isConcurrencySafe: () => true` 才能被并行调度

类比：机场安检中默认假设所有行李需要检查（fail-closed），只有经认证才能走快速通道。

### 45+ 核心工具分类

| 类别 | 工具 | 并发安全 |
|------|------|---------|
| 执行 | BashTool | 否（副作用） |
| 文件 | FileReadTool, FileEditTool, FileWriteTool | Read 是，Edit/Write 否 |
| 搜索 | GlobTool, GrepTool | 是 |
| 网络 | WebFetchTool, WebSearchTool | 是 |
| 智能 | AgentTool（子智能体入口） | 否 |
| 技能 | SkillTool | 否 |
| MCP | ListMcpResourcesTool, ReadMcpResourceTool | 是 |
| 搜索发现 | ToolSearchTool（延迟工具发现） | 是 |

> 超过一半的工具标记为并发不安全——这反映了 Agent 系统中大多数操作都有副作用的现实。

### 延迟工具发现（ToolSearchTool）

当工具数量超过阈值时，不在初始系统提示中发送所有工具的完整 schema，只发送工具名称列表，模型通过 ToolSearchTool 按需加载。

类比：传统方式把整本百科全书放在模型面前；延迟发现给模型一个目录索引。

MCP 工具总是延迟；`alwaysLoad` 标记的工具（如 AgentTool）不延迟。

### 工具过滤管线

```
getAllBaseTools()
  → 模式过滤（简单/普通模式：simple 只保留 Bash/Read/Edit）
  → 拒绝规则过滤（移除 blanket deny 工具）
  → 启用状态检查
  → 工具池组装（合并内建 + MCP，按名称排序去重）
  → 发送给 API 的工具列表
```

排序的目的：确保 prompt 缓存稳定性——工具顺序变化会导致缓存失效。

### 并发分区调度

`runTools` 函数基于**并发分区（concurrency partitioning）**策略：
1. 将工具调用按顺序划分为批次
2. 每个批次：一组连续的并发安全工具 OR 一个单独的非安全工具
3. 并发安全批次并行执行；非安全批次串行执行

示例：`[Read(a.ts), Read(b.ts), Bash(ls), Read(c.ts)]` → 
- Batch 1（并行）：Read(a.ts), Read(b.ts)
- Batch 2（串行）：Bash(ls)
- Batch 3（并行）：Read(c.ts)

并发度上限默认 10，由环境变量控制。

### StreamingToolExecutor 四阶段状态机

在模型还在生成 tool_use 块时就立即启动工具执行（流水线并发）。

每个工具的状态机：
```
queued → executing → completed → yielded
```

- **queued**：等待执行条件满足
- **executing**：检查并发条件后启动（无工具在执行，或所有执行中工具都并发安全）
- **completed**：执行完成，等待轮到输出
- **yielded**：结果已产出，生命周期结束

四个关键设计决策：
1. **顺序保证**：即使并行执行，结果 yield 仍保持请求顺序
2. **错误传播**：BashTool 失败取消所有并行兄弟；非 Bash 工具错误不传播
3. **进度即时产出**：进度消息绕过顺序约束，立即 yield 给 UI
4. **信号层级化**：每个工具使用独立子取消控制器，形成层级取消信号链

性能效益：相比传统批量执行，StreamingToolExecutor 可将多工具场景的总执行时间减少 50% 以上。

### 核心工具设计要点

**BashTool**：
- 执行失败时取消所有并行 Bash 工具（"快速失败"原则）
- 对命令进行 AST 解析 + 语义分析，识别搜索/读取操作用于 UI 折叠
- 可自定义中断行为（阻塞 vs 取消）

**FileEditTool**：
- 使用 `old_string → new_string` 精确替换模式，而非行号
- 行号是脆弱的（文件变化后行号偏移），字符串匹配是幂等的

**AgentTool（子智能体入口）**：
- 通过 `createSubagentContext` 创建独立 `ToolUseContext`，继承权限规则但拥有独立消息列表
- "继承但不共享"：子智能体不会意外修改父智能体状态
