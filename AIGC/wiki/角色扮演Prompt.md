# 角色扮演 Prompt

## 定义

通过为 Claude Code 设定具体的专家角色（而非泛泛的"你是专家"），将其从默认的"聪明实习生"模式升级为"专家顾问"模式。核心是三层结构：**身份层 → 经历层 → 行为约束层**，缺一不可。

## 核心观点

- **Claude Code 默认行为是"接单模式"**——给什么做什么，不会主动帮你想没说的（来源：[[角色扮演Prompt]]）
- 同样的任务，加角色设定前后代码质量可差 3 个档次（实验：用户认证模块，无角色用 MD5，有角色主动换 bcrypt + refresh token 轮换 + rate limiting）
- **"够用"与"生产级"之间有深鸿沟**：多时区、Stripe 幂等性、GDPR 合规、并发事务——"够用"可能意味着凌晨三点的告警

## 三层结构

### 身份层（你是谁）

告诉 Claude 具体的专精方向，不能只说"专家"。

- ❌ 弱：`你是一个 Node.js 专家`
- ✅ 强：`你是一个有 8 年经验的 Node.js 后端工程师，主要方向是 SaaS 产品的 API 设计和生产环境性能优化`

关键：工作方向决定关注点。"后端工程师"给通用解法，"专注 SaaS API 的后端工程师"自然考虑 multi-tenancy、rate limiting、webhook 可靠性。

### 经历层（假简历）

给 Claude "踩坑经历"，激活训练数据中与这些经历相关的知识权重。

> 你曾经处理过 Stripe 支付在高并发场景下的重复扣款问题，深知幂等性设计的重要性。你也在生产环境里见过时区处理不当导致的数据错乱。

原理：Claude 训练数据包含大量"专家踩坑经历"文本，给出具体背景时会自动调用相关知识权重——本质是"热身"，激活最相关的知识子集。

### 行为约束层（最关键）

没有这层，Claude 默认"你让我做什么我做什么"，不会主动暴露问题。

> 当你发现代码存在安全隐患、性能瓶颈或不符合生产标准的写法时，必须主动指出，不能只实现功能。如果需求本身存在设计问题，先说明问题再给出实现。

效果：从"执行者"变成"挑剔的同事"——功能实现了，但会主动说"这里的密码存储方式有安全风险"。

## 场景化专家模板

全局角色定通用准则，具体任务用更专精的角色：

| 场景 | 角色设定要点 |
|------|-------------|
| Stripe 支付 | Idempotency Key、Webhook 幂等性（先记录 event_id 再处理）、支付状态机单向流转、金额用整数（分） |
| 代码 Review | 安全漏洞 + 性能问题 + 可维护性 + 出海特有问题（时区/编码/国际支付），按 P0/P1/P2 严重程度分级 |
| 数据库设计 | 高频查询索引、软删除部分索引、TIMESTAMPTZ 带时区、N+1 查询风险 |

## CLAUDE.md 实践要点

- 角色设定放在 `CLAUDE.md` 的 project 级配置，Claude Code 每次启动自动读取
- **篇幅控制在 80 行以内**：超长 system prompt 会导致"注意力稀释"——重要规则和不重要规则混在一起，遵守率反而下降
- **当"核心价值观"写，不要当"操作手册"写**：3 条核心规则比 50 条操作细则遵守率更高
- 最关键的安全规则用 `**加粗**` 标注，提高优先级识别

> ⚠️ 作者踩坑：CLAUDE.md 写到 300 行后，明确写了"密码存储只使用 bcrypt"，Claude 仍用了 `crypto.createHash('sha256')`。精简到 80 行 + 加粗关键规则后问题消失。

## 完整模板示例（出海 SaaS）

```
# 角色定义
你是一位有 10 年经验的全栈工程师，专注于面向海外用户的 SaaS 产品开发。

## 技术背景
- 主要技术栈：Next.js / Node.js / PostgreSQL / Redis
- 擅长领域：支付集成（Stripe）、用户认证（JWT + Refresh Token）、多租户架构
- 曾处理过：Stripe Webhook 重复消费问题、多时区数据一致性问题、PostgreSQL 高并发写入的死锁问题

## 工作原则
**代码质量**
- 所有涉及用户数据的操作必须考虑事务安全
- 密码存储只使用 bcrypt（cost factor >= 12）
- 涉及金额的字段使用整数（分）存储，禁止浮点数

**安全默认值**
- JWT 过期时间：access token 15分钟，refresh token 7天
- 所有外部输入必须校验，使用 zod 或类似库
- SQL 查询禁止字符串拼接，只使用参数化查询

**主动暴露问题**
- 如果需求实现可能引入安全漏洞，必须先说明再实现
- 如果发现性能问题（如 N+1 查询），主动指出并给出优化方案
```

## 核心洞察

> **AI 是思维的放大器，不是思维的替代品。你设定的角色有多专业，它给你的答案就有多专业。**

把"经验"翻译成**具体的技术决策偏好**——不是"我经验丰富"，是"我倾向于在任何数据库操作里加事务，即使看起来不需要"。具体到行为层面，Claude 才能模拟。
