# 上下文四级压缩策略

## 定义

Claude Code 的上下文管理采用四级渐进式压缩策略，从零成本到高成本依次递进。核心哲学：不是"暴力截断"，而是"分级精细管理"——先裁剪旧工具结果，再微压缩过期缓存，再主动重构上下文，最后才做全量 LLM 摘要。

## 有效窗口公式

```
有效窗口 = 模型窗口 - 预留输出令牌
```

`getEffectiveContextWindowSize` 函数：预留 min(最大输出令牌, 20,000) 作为压缩摘要的输出空间。

示例（200K 模型）：
- 预留空间 = min(16,384, 20,000) = 16,384
- 有效窗口 = 200,000 - 16,384 = **183,616 令牌**

**为什么预留 20,000？** 防止"压缩悖论"——压缩操作本身需要输出令牌，若不预留，压缩可能因空间不足而失败。

## 四级安全阈值

| 区域 | 令牌使用比例 | 触发行为 |
|------|-----------|---------|
| 安全区域 | 0% - 85% | 正常工作，无需干预 |
| 警告区域 | 85% - 90% | 显示黄色警告（WARNING） |
| 危险区域 | 90% - 95% | 自动压缩触发（AUTOCOMPACT） |
| 阻塞区域 | 95% - 100% | 拒绝新请求（BLOCKING） |

关键常量：
- `AUTOCOMPACT_BUFFER_TOKENS`：13,000（自动压缩触发缓冲）
- `WARNING_THRESHOLD_BUFFER_TOKENS`：20,000（早期预警）
- `MANUAL_COMPACT_BUFFER_TOKENS`：3,000（手动压缩最小余量）

## 四级压缩策略详解

### Level 1：Snip（裁剪）

**成本：≈ 0（无 LLM 调用）**

将旧的工具调用结果替换为标记文本 `[Old tool result content cleared]`，而非删除消息——删除会破坏消息链连续性（后续消息可能引用工具调用 ID）。

**触发方式**：用户手动标记。适用于：读取了大量文件分析完成后，立即回收空间。

### Level 2：MicroCompact（微压缩）

**成本：极低（无 LLM 调用，字符串替换）**

**触发条件**：距上次助手消息时间超过配置阈值（服务端缓存已过期）。逻辑：既然缓存过期无论如何都要重传，不如主动清除旧工具结果降低重传负载。

保留最近 N 个可压缩工具结果（`config.keepRecent`，最小值为 1），清除其余。可压缩工具类型：Read、Bash、Grep、Glob、WebSearch、WebFetch、Edit、Write。

还有一个高级路径：通过 API `cache_edits` 机制无损删除工具结果而不破坏缓存前缀。

### Level 3：Collapse（折叠）

**成本：中等（部分 LLM 调用）**

**触发条件**：上下文利用率达 90% 时开始提交压缩操作，95% 时阻止新的 spawn。

理念：从"被动压缩"转变为"主动重构"。选择性重构消息组（vs AutoCompact 的全对话摘要），保留更多原始细节。

Collapse 比 AutoCompact 优先级更高，在 93% 临界点会抑制 AutoCompact 触发。

### Level 4：AutoCompact（自动压缩）

**成本：高（完整 LLM API 调用）**

**触发条件**：令牌用量超过自动压缩阈值，且前三级无法释放足够空间。最后的兜底手段。

压缩流程：
1. 执行 PreCompact 钩子（可注入自定义压缩指令，可阻止压缩）
2. 选择压缩提示模板（BASE / PARTIAL / PARTIAL_UP_TO）
3. 流式生成摘要（forked agent 或主线程，最多 1 轮，禁止工具调用）
4. 若出现 `prompt-too-long` 错误，截断最老的 API 轮次组并重试（最多 3 次）
5. 重建上下文：CompactBoundaryMessage + 摘要 + 附件 + 钩子结果
6. 执行 PostCompact 钩子

## 断路器保护

**问题**：若压缩持续失败（网络波动、API 错误），盲目重试会造成 API 调用雪崩。

**解决方案**：断路器（Circuit Breaker）模式。连续失败次数达 3 次（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES`）后停止尝试压缩，成功后计数器归零。

**真实数据**：引入前曾观察到 1,279 个会话出现 50 次以上的连续压缩失败（最高 3,272 次），每天浪费约 250K 次 API 调用。引入断路器后彻底消除。

## 压缩提示工程

### 三种压缩模板

| 模板 | 场景 | 压缩范围 |
|------|------|---------|
| BASE_COMPACT_PROMPT | 常规自动压缩 | 从对话开始到当前所有消息 |
| PARTIAL_COMPACT_PROMPT | 对话前半段已被压缩过 | 从指定消息到当前 |
| PARTIAL_COMPACT_UP_TO_PROMPT | 保留最近完整消息 | 从对话开始到指定消息 |

每个模板都包含**防工具调用前导指令**：要求模型仅以文本形式回复，不得调用任何工具。

### 双阶段输出结构

```
LLM 输出 = <analysis> 思维草稿本 + <summary> 正式摘要
```

- `<analysis>` 块：Chain-of-Thought 载体，提升摘要质量，**在最终上下文中被丢弃**（节省令牌）
- `<summary>` 块：包含 9 个结构化章节，进入新的上下文

**设计哲学：思考是过程，摘要是结果。过程不计费，结果才计入上下文。**

### CompactBoundaryMessage（压缩边界标记）

每次压缩后插入消息流中，携带元数据：触发类型（手动/自动）、压缩前令牌数、涉及消息数量。`logicalParentUuid` 字段构建逻辑连续性，防止后续压缩重复压缩已摘要内容。

## 压缩后令牌预算

| 常量 | 值 |
|------|-----|
| POST_COMPACT_TOKEN_BUDGET | 50,000（总预算上限） |
| POST_COMPACT_MAX_FILES_TO_RESTORE | 5（最多恢复文件数） |
| POST_COMPACT_MAX_TOKENS_PER_FILE | 5,000（每文件令牌上限） |
| POST_COMPACT_SKILLS_TOKEN_BUDGET | 25,000（技能独立预算） |

**反模式**：压缩后立即重新加载所有文件 → "压缩-膨胀-再压缩"恶性循环。正确做法：只重新加载当前任务需要的文件。

## 长对话上下文管理策略

| 场景 | 推荐策略 |
|------|---------|
| 阅读 10+ 文件后 | Snip 清除已分析完的文件内容 |
| 长时间暂停后返回 | MicroCompact 自动清除过期缓存 |
| 连续实施多个功能 | 每完成一个功能后手动 /compact |
| 跨多个子系统重构 | 分阶段工作 + 记忆系统辅助 |

**主动压缩优于被动压缩**：在 70% 时手动 `/compact` 并附带保留提示，比等到 95% 自动触发能保留更多有价值信息。

**分阶段工作**：研究阶段 → 压缩 → 规划阶段 → 压缩 → 实施阶段，每阶段结束时压缩确保下阶段有充足空间。
